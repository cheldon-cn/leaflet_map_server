# ArcGIS Pro 工程迁移工具需求规格说明书

## 1. 项目概述

本项目旨在开发一款自动化迁移工具，支持将 ArcGIS Pro 2.x/3.x 工程文件（.aprx）及其关联数据、地图、符号、布局等完整内容，高效、准确地迁移至通用 GIS 数据模型（参考 QGIS 3.34 LTS 模型）。工具需支持批量处理、异步执行、多线程优化，并针对超大数据库表提供高性能复制方案。

## 2. 迁移目标与范围

### 2.1 核心目标
- **一键迁移**：支持 ArcGIS Pro 2.x 和 3.x 工程文件（.aprx）的完整迁移。
- **数据同步**：同步迁移工程中的地图、图层、符号、标注、布局等所有可视化与配置元素。
- **模型转换**：迁移目标为通用 GIS 数据模型，可参考 QGIS 模型，确保在 QGIS 3.34 LTS 中能够正确重建。

### 2.2 迁移范围
- **单个工程迁移**：完整解析 .aprx 文件，提取所有相关信息。
- **批量处理**：支持同时处理多达 **200 个**工程文件。
- **数据类型**：
  - 本地矢量/栅格数据（如 Shapefile、File Geodatabase 等）。
  - PostgreSQL/PostGIS 数据库中的空间表、视图、函数、触发器。
  - 自定义样式文件（.stylx）。
  - 嵌入的 Python 脚本、地理处理工具箱。

## 3. 性能要求

- **处理效率**：在保证正确性的前提下，**单个工程的迁移总时间应控制在 30 秒以内**。
- **主要耗时点**：
  - PyQGIS 读取 XML 并重建工程的环节。
  - PostgreSQL 大表的导出（pg_dump）。
- **优化方向**：对重建环节进行多线程/多进程优化，参考 QgsTask 设计并行重建方案。

## 4. 系统架构设计

### 4.1 两阶段解耦架构
1. **读取与中间文件生成阶段**（在“迁移工作站”执行）：
   - 需要在一台安装了 **ArcGIS Pro 2.x/3.x** 的专用机器（迁移工作站）上运行。
   - 使用 **Python + ArcPy** 解析 .aprx 工程，提取所有元数据、配置、采样数据，生成标准化的中间 **XML 文件**。
   - 该阶段输出包含完整工程描述、数据采样记录、文件夹结构信息的 XML。

2. **分发与重建阶段**（可在无 ArcGIS Pro 环境执行）：
   - 将中间 XML 文件分发到任意目标机器（无需安装 ArcGIS Pro）。
   - 使用 **PyQGIS** 读取 XML，按照通用 GIS 模型（QGIS）重建工程、导入数据、恢复符号与布局。

### 4.2 异步任务管理
- 将**数据复制**与**工程转换**解耦，设计为独立的、可监控、可重试的后台任务。
- 中间 XML 中包含任务状态字段（pending、running、completed、failed），便于工作流引擎进行状态跟踪、重试与报警。

## 5. 中间 XML 格式规范

### 5.1 设计原则
- 定义完整的 **XML Schema（XSD）** 以确保数据结构的严格一致。
- 尽可能保留原始工程信息，确保在重建时不遗漏任何配置。

### 5.2 主要元素（XSD 结构）
| 元素 | 描述 |
|------|------|
| **工程元数据** | 版本、保存时间、原始工程主文件夹路径。 |
| **地图列表** | 每个地图包含名称、空间参考、图层列表。 |
| **图层定义** | 数据源路径/连接字符串、CRS（完整 WKT 或 Proj 字符串）、字段定义（名称、类型、长度）、定义查询、标注设置、符号系统（CIM JSON 格式）。 |
| **布局定义** | 页面设置（大小、单位）、元素列表（地图框、图例、比例尺、文本、图片等），要求元素位置、大小、字体、边框等属性**像素级精确**。 |
| **样式库** | 从 .stylx 提取的符号定义，转换为 QGIS 样式库格式（.xml）。 |
| **脚本清单** | 嵌入的 Python 脚本/工具箱信息：名称、所属工具箱、输入参数、核心逻辑描述。 |
| **PostgreSQL 任务状态** | 记录每张表的数据复制状态（pending/completed/failed）。 |

### 5.3 数据采样记录
为每个图层预计算并存储以下采样信息，用于后续的自动化验证：
- **要素记录数**。
- **前 N 条记录（默认 N=10）的属性和几何采样**：
  - 属性值：包含所有字段（BLOB 除外）。
  - 几何类型：以 WKT/WKB 格式记录。
- **原始工程的文件夹结构**：辅助重建时组织数据路径。

### 5.4 动态内容与高级设置
- **CRS 预计算**：存储每个图层的完整 CRS WKT，避免重建时昂贵的查找操作。
- **动态文本**：支持日期、页码、元数据等动态标签的转换。
- **格网与经纬网**：建立详细的属性映射表，转换间隔、标注格式、线型等设置。

## 6. 数据迁移策略

### 6.1 PostgreSQL 数据复制
提供两种复制方式，根据表大小和业务需求自动选择：

| 方式 | 适用场景 | 实现方案 |
|------|----------|----------|
| **完全复制**（结构+数据） | 中小型表，或需要完全迁移的场景 | a. 提取表结构和数据，写入中间格式（如 GeoPackage）<br>b. 自动触发 PostgreSQL 原生工具（pg_dump / pg_restore） |
| **仅更新连接字符串** | 数据已存在于目标环境，仅需重定向连接 | 基于表名（schema.table）查询外部映射表，在生成中间 XML 时即解析好目标连接字符串，支持简单的字段替换和复杂的路由决策。 |

### 6.2 数据库对象迁移
- **空间函数/视图/触发器**：依赖特定 PostGIS 函数，提供两种迁移策略：
  1. **生成报告供 DBA 审核后手动执行**（半自动化）。
  2. **完全自动化迁移**（需评估函数兼容性）。
- **PostGIS 数据类型**：无需特殊处理，按通用 GIS 模型转换。

### 6.3 正确性判定标准
- **数据结构对比**：迁移前后图层的字段名、类型、长度完全一致。
- **功能对比**：布局导出效果、数据查询结果一致。

## 7. 样式与符号转换

- **源文件**：ArcGIS Pro 自定义样式文件（.stylx）。
- **目标格式**：QGIS 样式库格式（.xml）。
- **转换方法**：
  - 提取 .stylx 中的每个样式项（不包括 ArcGIS Pro 特有的符号效果及特殊自定义符号）。
  - 借鉴或集成开源社区现有成果（如 **SLYR 社区版**中关于 .style 转换的部分）。
- **输出**：将转换后的样式库嵌入中间 XML 的“样式库”元素中。

## 8. 脚本与工具箱处理

- **脚本语言**：主要针对工程中嵌入的 **Python** 脚本（如字段计算、缓冲区分析等简单地理处理脚本）。
- **转换策略**：
  - **不进行自动 ArcPy 语法转换**。
  - 提取脚本的**结构化清单**：脚本名称、所属工具箱、输入参数、核心逻辑描述。
  - 生成**待办清单**，为半自动化重建提供基础。
- **扩展接口**：预留接口，支持未来对工具箱或复杂脚本的进一步处理。

## 9. 多线程/多进程优化

### 9.1 任务拆分策略
1. **按工程拆分**：每个 .aprx 文件作为一个独立任务，并行处理多个工程。
2. **按工程内组件拆分**：将单个工程内的地图、布局进一步拆分为子任务，提高并发度。
3. **依赖关系处理**：对于有依赖关系的任务（如先复制数据再重建图层），按依赖关系合理调度。

### 9.2 并行框架
- 参考 **QgsTask** 实现，设计轻量级任务队列。
- 利用 Python 的 `concurrent.futures` 或多进程模块（`multiprocessing`）实现并行重建。

## 10. 超大表处理策略（>1TB）

### 10.1 阈值检测
- 在 ArcPy 读取阶段，检测 PostgreSQL 表的预估大小。
- 若表大小超过阈值（例如 **100GB**），则在中间 XML 中标记为 **“建议逻辑复制”**。

### 10.2 逻辑复制配置
- 在中间 XML 中嵌入逻辑复制的配置参数（如发布/订阅名称、筛选条件）。
- 生产环境允许迁移工具自动执行 DDL 并管理逻辑复制流程。

### 10.3 混合处理模式
- 同一工程中可能同时包含**小表**（适合 pg_dump）和**大表**（适合逻辑复制）。
- 工具应能自动识别并分别采用不同的复制方式，确保整体效率。

## 11. 测试与验证

### 11.1 自动化测试脚本
在 QGIS 端重建完成后，自动执行以下验证步骤：
1. **读取采样记录**：从中间 XML 中加载每个图层的采样数据（前 N 条记录）。
2. **执行相同查询**：对重建后的图层按相同条件（如按 OBJECTID 获取前 N 条）查询。
3. **对比结果**：
   - 属性值是否完全一致。
   - 几何 WKT 是否相同。
   - 空间参考是否一致。
4. **生成报告**：输出一致性报告，标记不一致的图层及其差异详情。

### 11.2 属性映射表
为每一个核心 GIS 对象（如图层符号、标注引擎、布局元素）创建详细的**属性映射表**，确保转换过程中属性一一对应，减少信息丢失。

## 12. 开发工具与环境

- **主要开发语言**：Python。
- **ArcGIS Pro 端**：使用 **ArcPy** 进行工程解析、元数据提取。
- **QGIS 重建端**：使用 **PyQGIS** 进行工程重建、数据加载。
- **第三方工具**：
  - PostgreSQL 工具链：pg_dump, pg_restore, 逻辑复制。
  - 样式转换：参考 SLYR 社区版（**不开源商业插件**，仅借鉴其思路）。
- **版本要求**：
  - ArcGIS Pro: 2.x / 3.x。
  - QGIS: 3.34 LTS。

## 13. 实施步骤建议

1. **环境准备**：在安装 ArcGIS Pro 2.x/3.x 的机器上搭建 Python 环境，熟悉 `arcpy.mp` 和 CIM 结构。
2. **调研现有工具**：下载 SLYR 社区版，使用典型工程测试其转换效果，重点关注符号转换准确度、布局元素完整性、PostgreSQL 连接处理方式。
3. **编写第一个原型**：实现最小可行的“读取+生成XML”脚本，功能包括：
   - 打开 .aprx 文件。
   - 列出所有地图和图层。
   - 输出简单 XML（包含图层名称、数据源路径）。
4. **设计 XML Schema 草案**：基于对 CIM 结构的初步了解，创建 XSD 文件。
5. **迭代开发**：逐步增加数据采样、样式转换、数据库复制等高级功能。
6. **性能优化**：引入多线程/多进程，针对超大表实现逻辑复制方案。
7. **测试与验证**：开发自动化测试脚本，确保迁移结果的正确性。
