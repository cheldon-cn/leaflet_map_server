# 多Skill协同处理机制分析

**版本**: v1.0
**日期**: 2026-04-03
**分析对象**: flow-demand-to-design 及其关联Skill

---

## 一、概述

本文档分析项目中多Skill协同处理的触发条件、调用关系和协调机制，以flow-demand-to-design为核心案例，探讨如何保证单个Skill的稳定高效工作，同时确保多Skill的有效协调。

## 二、核心Skill触发条件分析

### 2.1 Skill触发条件汇总表

| Skill名称 | 触发关键词 | 触发优先级 | 独立使用 | 被编排调用 |
|-----------|-----------|-----------|---------|-----------|

### 2.2 触发条件详细分析

#### 2.2.1 flow-demand-to-design（流程编排Skill）

**触发场景**:
```
用户输入: "将需求文档转换为设计文档"
用户输入: "转换 xxx_demand.md 为设计文档"
用户输入: "需要从需求到设计的完整流程"
```

**触发判断逻辑**:
```
IF 用户输入包含 ("需求转设计" OR "需求到设计" OR "完整流程" OR "端到端转换")
THEN
  → 触发 flow-demand-to-design
  → 自动编排5个子步骤
END IF
```

**特点**:
- 作为编排型Skill，不直接执行具体任务
- 负责协调其他Skill的调用顺序
- 处理步骤间的数据传递和状态管理

#### 2.2.2 demand-analyzer（需求分析Skill）

**触发场景**:
```
独立触发:
  用户输入: "分析需求文档 xxx.md"
  用户输入: "验证需求是否完整"
  用户输入: "生成需求规格说明书"

被编排调用:
  flow-demand-to-design Step 1
```

**触发判断逻辑**:
```
IF 用户输入包含 ("需求分析" OR "需求验证" OR "需求提取" OR "需求规格说明书")
   OR 被flow-demand-to-design调用
THEN
  → 触发 demand-analyzer
END IF
```

#### 2.2.3 design-generator（设计生成Skill）

**触发场景**:
```
独立触发:
  用户输入: "基于需求规格生成设计文档"
  用户输入: "设计架构"

被编排调用:
  flow-demand-to-design Step 2
```

**触发判断逻辑**:
```
IF 用户输入包含 ("设计文档" OR "架构设计" OR "模块划分" OR "生成设计")
   AND 存在需求规格说明书
   OR 被flow-demand-to-design调用
THEN
  → 触发 design-generator
END IF
```

#### 2.2.4 design-doc-validator（设计验证Skill）

**触发场景**:
```
独立触发:
  用户输入: "检查设计文档质量"
  用户输入: "验证设计文档是否合格"

被编排调用:
  flow-demand-to-design Step 3
```

**触发判断逻辑**:
```
IF 用户输入包含 ("文档质量" OR "预处理" OR "检查清单" OR "验证文档")
   OR 被flow-demand-to-design调用
THEN
  → 触发 design-doc-validator
END IF
```

#### 2.2.5 multi-role-reviewer vs technical-reviewer（评审Skill选择）

**选择决策树**:
```
用户输入: "评审xxx文档"
       │
       ▼
┌─────────────────────────────────────┐
│ 包含 "交叉评审" OR "多角色"?         │
├─────────────────────────────────────┤
│         │                           │
│    YES  │   NO                      │
│         │                           │
│    ▼    │    ▼                      │
│ multi-  │  包含特定专家角色?          │
│ role-   │    │                      │
│ reviewer│ YES│   NO                 │
│         │    ▼    ▼                 │
│         │ tech- 默认使用             │
│         │ nical- multi-role         │
│         │ reviewer                  │
└─────────────────────────────────────┘
```

**优先级规则**:
| 优先级 | Skill | 条件 |
|--------|-------|------|
| 1 | 用户指定 | 用户明确指定Skill名称 |
| 2 | multi-role-reviewer | 包含"交叉评审"、"多角色"、"投票" |
| 3 | technical-reviewer | 包含"评审"、"审计"但无多角色关键词 |

## 四、单Skill稳定高效工作机制

### 4.1 单Skill设计原则

#### 4.1.1 输入验证机制

每个Skill在执行前必须进行输入验证：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      输入验证流程                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Step 0: 输入验证                                                       │
│   ├── 验证输入非空                                                       │
│   ├── 验证输入格式                                                       │
│   ├── 验证输入大小                                                       │
│   └── 验证输入完整性                                                     │
│                                                                         │
│   IF 验证失败 THEN                                                       │
│       → 返回错误报告                                                     │
│       → 不执行后续步骤                                                   │
│   END IF                                                                │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**验证规则示例**（以demand-analyzer为例）:

| 验证项 | 通过条件 | 失败处理 |
|--------|----------|----------|
| 文档非空 | 文档长度 > 0 | 返回错误：文档为空 |
| 文档格式 | 可解析为文本 | 返回错误：格式不支持 |
| 文档大小 | < 1MB | 提示：文档过大，建议拆分 |

#### 4.1.2 输出质量保证

每个Skill必须定义输出质量标准：

**demand-analyzer输出质量标准**:
```
□ 每个需求有唯一ID (F001, F002...)
□ 每个需求有验收标准
□ 优先级已分配 (MoSCoW)
□ 冲突已检测
□ 需求来源已标注
```

**design-generator输出质量标准**:
```
□ 100%功能需求有设计对应
□ 每个模块职责清晰
□ 每个接口有明确输入输出
□ 技术选型有理由
```

#### 4.1.3 错误处理机制

**错误分级**:

| 级别 | 说明 | 处理策略 |
|------|------|----------|
| 🔴 致命 | 无法继续处理 | 终止处理，返回错误报告 |
| 🟠 严重 | 部分功能受影响 | 跳过问题项，继续处理 |
| 🟡 警告 | 不影响主流程 | 标记警告，继续处理 |
| 🔵 信息 | 提示性信息 | 记录日志 |

**错误处理流程**:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      错误处理流程                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   TRY                                                                   │
│       执行Skill核心逻辑                                                  │
│   CATCH 致命错误 THEN                                                    │
│       → 终止执行                                                        │
│       → 返回错误报告                                                     │
│       → 建议用户操作                                                    │
│   CATCH 严重错误 THEN                                                    │
│       → 记录错误                                                        │
│       → 跳过问题项                                                      │
│       → 继续执行                                                        │
│   CATCH 警告 THEN                                                        │
│       → 记录警告                                                        │
│       → 继续执行                                                        │
│   END TRY                                                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 4.1.4 性能保障

**性能预期定义**:

| Skill | 小型输入 | 中型输入 | 大型输入 | 超大型输入 |
|-------|---------|---------|---------|-----------|
| demand-analyzer | 10-30秒 | 30-60秒 | 1-2分钟 | 2-5分钟 |
| design-generator | 30-60秒 | 1-2分钟 | 2-5分钟 | 5-10分钟 |
| design-doc-validator | 10-30秒 | 30-60秒 | 1-3分钟 | 3-5分钟 |
| multi-role-reviewer | 1-2分钟 | 2-5分钟 | 5-10分钟 | 10-20分钟 |

**性能优化策略**:
1. 分批处理大型输入
2. 缓存中间结果
3. 并行处理独立任务
4. 设置超时限制

### 4.2 单Skill约束清单

**必须做的事**:
- [ ] 执行输入验证
- [ ] 遵循输出质量标准
- [ ] 处理所有错误级别
- [ ] 记录执行日志
- [ ] 返回结构化结果

**不要做的事**:
- [ ] 不要跳过输入验证
- [ ] 不要修改用户原始数据意图
- [ ] 不要添加未提及的功能
- [ ] 不要忽略错误
- [ ] 不要无限等待

## 五、多Skill有效协调机制

### 5.1 编排层设计

flow-demand-to-design作为编排层Skill，负责：

#### 5.1.1 步骤顺序控制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      步骤顺序控制                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Step 1: demand-analyzer                                               │
│   │   条件: 存在需求文档                                                 │
│   │   输出: 需求规格说明书                                               │
│   │                                                                     │
│   ▼ (必须等待Step 1完成)                                                │
│   Step 2: design-generator                                              │
│   │   条件: Step 1输出存在                                               │
│   │   输出: 设计文档初稿                                                 │
│   │                                                                     │
│   ▼ (必须等待Step 2完成)                                                │
│   Step 3: design-doc-validator                                          │
│   │   条件: Step 2输出存在                                               │
│   │   输出: 验证报告                                                     │
│   │                                                                     │
│   ▼ (必须等待Step 3通过)                                                │
│   Step 4: multi-role-reviewer                                           │
│   │   条件: Step 3验证通过                                               │
│   │   输出: 评审报告、评分                                               │
│   │                                                                     │
│   ▼ (根据评分决定)                                                       │
│   Step 5: 迭代改进                                                       │
│       IF 评分 < 95 THEN                                                  │
│           → 修改设计文档                                                 │
│           → 重新验证 (Step 3)                                           │
│           → 重新评审 (Step 4)                                           │
│           → 重复直到评分≥95 或 迭代次数>6                                │
│       END IF                                                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 5.1.2 数据传递机制

**文件传递方式**:

```
Step 1 输出                    Step 2 输入
┌─────────────────┐           ┌─────────────────┐
│ xxx_requirement │ ────────► │ design-generator│
│ _spec.md        │           │ 读取该文件       │
└─────────────────┘           └─────────────────┘
```

**传递数据结构**:

| 步骤 | 输出文件 | 传递内容 |
|------|----------|----------|
| Step 1 | xxx_requirement_spec.md | 功能需求、非功能需求、优先级、追溯矩阵 |
| Step 2 | xxx_design.md | 架构、模块、接口、追溯矩阵 |
| Step 3 | 验证报告 | 检查结果、问题列表 |
| Step 4 | 评审报告 | 评分、改进建议 |

#### 5.1.3 状态管理

**状态定义**:

| 状态 | 说明 | 后续动作 |
|------|------|----------|
| pending | 步骤未开始 | 等待前置步骤完成 |
| running | 步骤执行中 | 等待执行完成 |
| success | 步骤成功 | 继续下一步骤 |
| failed | 步骤失败 | 错误处理或回退 |
| retry | 需要重试 | 重新执行当前步骤 |

**状态转换图**:

```
┌─────────┐     开始执行     ┌─────────┐
│ pending │ ───────────────►│ running │
└─────────┘                 └────┬────┘
                                 │
                    ┌────────────┼────────────┐
                    │            │            │
                    ▼            ▼            ▼
              ┌─────────┐  ┌─────────┐  ┌─────────┐
              │ success │  │ failed  │  │ retry   │
              └────┬────┘  └────┬────┘  └────┬────┘
                   │            │            │
                   ▼            ▼            │
              下一步骤      错误处理      重新执行
                   │            │            │
                   │            ▼            │
                   │      ┌─────────┐        │
                   │      │ 回退或  │        │
                   │      │ 终止    │        │
                   │      └─────────┘        │
                   │                         │
                   └─────────────────────────┘
```

### 5.2 回退机制

**回退策略**:

| 场景 | 回退策略 | 说明 |
|------|----------|------|
| Step 1失败 | 终止流程 | 需求文档问题，无法继续 |
| Step 2失败 | 回退Step 1 | 补充需求信息后重新设计 |
| Step 3失败 | 回退Step 2 | 修改设计后重新验证 |
| Step 4评分<95 | 进入Step 5 | 迭代改进 |
| Step 5超限 | 终止流程 | 保留产物，请求人工介入 |

**回退流程图**:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      回退机制流程                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Step 1 失败                                                           │
│   │                                                                     │
│   └──► 终止流程                                                         │
│        └──► 返回需求文档问题报告                                         │
│                                                                         │
│   Step 2 失败                                                           │
│   │                                                                     │
│   └──► 回退到 Step 1                                                    │
│        └──► 补充需求信息                                                 │
│        └──► 重新执行 Step 2                                             │
│                                                                         │
│   Step 3 失败                                                           │
│   │                                                                     │
│   └──► 回退到 Step 2                                                    │
│        └──► 修改设计文档                                                 │
│        └──► 重新执行 Step 3                                             │
│                                                                         │
│   Step 4 评分 < 95                                                       │
│   │                                                                     │
│   └──► 进入 Step 5 迭代改进                                              │
│        └──► 修改设计 → Step 3 → Step 4                                  │
```

### 5.4 超时处理机制

**超时类型与处理**:

| 超时类型 | 检测时机 | 处理策略 |
|----------|----------|----------|
| 单次迭代超时 | 每次迭代开始时记录时间 | 跳过当前迭代，记录日志，继续下一次 |
| 总迭代超时 | 每次迭代结束时检查总时间 | 终止迭代，保留产物，生成超时报告 |
| 单Skill超时 | Skill执行开始时记录时间 | 终止Skill，返回部分结果，记录超时 |

**超时报告格式**:

```markdown

**已执行时间**: [时间]
**当前步骤**: [步骤名称]
**当前评分**: [分数]

**已生成产物**:
- [产物列表]

**建议**:
1. 检查输入文档是否有过于复杂的描述
2. 考虑拆分大型输入文档

```

## 七、最佳实践建议

### 7.1 单Skill使用建议

1. **输入准备**: 确保输入文档格式正确、内容完整
2. **预期管理**: 了解Skill的性能预期，合理安排时间
3. **结果验证**: 检查输出是否符合质量标准
4. **问题反馈**: 遇到问题时提供详细的错误信息

### 7.2 多Skill协调建议

1. **流程完整性**: 使用flow-demand-to-design确保流程完整
2. **产物保留**: 保留所有中间产物，便于追溯和回退
3. **迭代控制**: 设置合理的迭代次数和超时限制
4. **人工介入**: 当自动流程无法达标时，及时请求人工介入

### 7.3 错误处理建议

1. **查看错误报告**: 仔细阅读错误报告中的建议
2. **按优先级修复**: 先修复致命错误，再处理警告
3. **增量验证**: 修复后增量验证，避免全量重跑
4. **日志分析**: 查看执行日志，定位问题根因

## 八、总结

### 8.1 核心机制总结

| 机制 | 说明 | 关键点 |
|------|------|--------|
| 触发条件 | 基于关键词匹配和上下文推断 | 优先级规则、决策树 |
| 调用关系 | 编排层协调多个执行层Skill | 顺序控制、数据传递 |
| 单Skill保障 | 输入验证、输出标准、错误处理 | 质量标准、性能预期 |
| 多Skill协调 | 状态管理、回退机制、迭代控制 | 超时处理、产物保留 |

### 8.2 设计原则

1. **单一职责**: 每个Skill只负责一个明确的功能
2. **接口清晰**: Skill间通过明确的接口和数据传递
3. **错误隔离**: 单个Skill的错误不影响其他Skill
4. **可观测性**: 完整的日志和状态追踪
5. **可恢复性**: 支持回退和重试机制

### 8.3 改进方向

1. **并行执行**: 对于无依赖的步骤，支持并行执行
2. **增量处理**: 支持增量更新，避免全量重跑
3. **智能推荐**: 根据上下文智能推荐下一个Skill
4. **可视化监控**: 提供流程执行的可视化监控界面
