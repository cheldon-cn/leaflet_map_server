# 海图显示导航系统需求原始梳理

## 一、需求汇总

### 1.1 系统定位

开发一套跨平台海图显示导航系统，主要应用于内河航道、湖泊、入海口、近海区域，辅助船舶安全高效行驶。

**原始需求描述：**
> "我想生成开发一个海图显示导航系统，主要应用与内河航道、湖泊、入海口、近海区域；海图数据包括S57、s101及其他s100系列数据，要求有桌面端、移动端、web端；一套跨平台内核，多端使用；实时渲染要求百万级数量<300ms,依赖北斗定位信息，实时获取位置信息，根据位置显示高精地图（精度<1m）根据实时水深制定航线，定时更新航道信息，推送航道预警信息，辅助船舶安全高效行驶；"

### 1.2 核心需求概览

| 需求类别 | 核心内容 | 优先级 |
|---------|---------|-------|
| 数据支持 | S57、S101及S100系列数据 | P0 |
| 平台支持 | 桌面端(Windows/Linux)、移动端(Android)、Web端 | P0 |
| 性能要求 | 百万级要素渲染<300ms | P0 |
| 定位系统 | 北斗定位，精度<1m | P1 |
| 航线规划 | 多因素智能规划 | P1 |
| 预警推送 | 多类型预警，提前30min | P1 |
| 离线能力 | 各端支持离线使用 | P1 |
| 模拟模式 | 航行模拟、预警演练 | P2 |

### 1.3 需求澄清问答

**Q1: 数据来源与处理**
- 数据格式：需将海图数据转换为内部格式，没有现成数据服务器，正在开发内部数据API
- 更新频率：包括每日、实时、不更新；以主动推送更新为主，进行增量更新；此功能可为VIP功能
- 覆盖范围：初期覆盖长江航道武汉段，后期扩展至珠江流域、洞庭湖、太湖及东南沿海区域，再扩展至全球
- 水深数据：主要基于已有海图数据库，VIP支持实时采集

**Q2: 用户规模与平台**
- 并发用户：预计同时支持1000用户并发使用
- 桌面端：Windows/Linux
- 移动端：Android
- Web端：浏览器
- 离线支持：各端支持离线使用，对外功能保持一致
- 内核技术：C++核心逻辑 + Java API，其他技术可做对比参考选型

**Q3: 性能指标细化**
- "百万级数量"：是指渲染所有要素数量
- 300ms：为从数据加载到完全显示的时间
- 测试环境：
  - Windows：Intel i5 8265/16G内存/2G显卡
  - 移动端：2.6GHz八核处理器/6G内存
- 高精度地图<1m：为定位精度，需要配备相应硬件

**Q4: 航线规划与预警**
- 规划因素：结合水深、航道宽度、桥梁限高等多因素进行全局路径规划
- 动态避障：需要考虑船舶吃水深度、动态避障
- 预警类型：水深突变、恶劣天气、船只碰撞风险、航道关闭等
- 阈值配置：部分阈值由系统预设，部分由用户自定义
- 推送方式：应用内通知、短信、声音警报（VIP功能）
- 更新方式：系统自动从外部源获取更新（VIP），同时允许管理员手动录入

**Q5: 北斗定位集成**
- 设备输出格式：NMEA 0183/2000
- 数据获取方式：串口和系统API获取
- 用户类型区分：
  - 普通用户：通过手机定位获取定位数据
  - VIP用户：通过专用设备串口获取
- 定位更新频率：
  - VIP用户：10秒更新1次
  - 普通用户：30秒1次

**Q6: 坐标系与显示**
- 坐标系支持：WGS-84、CGCS2000；也支持其他坐标系，支持不同坐标系间转换
- 前期目标：以显示基础海图为主，支持图层开关、缩放等级控制
- 后期目标：叠加AIS目标、气象图层等图层
- 交互方式：触摸手势（移动端）与鼠标键盘（桌面）不需统一，各自发挥自己的优势；但需保持主要功能的一致性

**Q7: 多语言与模式**
- 语言支持：中英文界面，支持扩展西班牙语、拉丁语、东南亚语种等
- 模式支持：航行、路径规划、预警等支持开启模拟模式和实际工作模式

**Q8: 安全与权限**
- 合规要求：需要符合国家地理信息保密规定，需考虑支持加密存储和传输，预留接口，后期实现
- 权限划分：不同角色（船员、调度中心、管理员）的权限划分，看到的功能不一样，可操作的也不一样；待细分
- 冗余设计：系统需要有冗余设计，增强鲁棒性，稳定可靠性
- 预警响应时间：提前30min，不同级别预警时间不一，但至少要提前30s

**Q9: 开发资源**
- 前期目标：显示基础海图，以3个月为限
- 代码库：有部分代码库可复用，大部分依赖AI
- 团队能力：熟悉GIS开发，了解海图相关解析显示，航海电子技术了解不多

### 1.4 开发阶段规划

| 阶段 | 时间 | 目标 | 覆盖范围 |
|-----|-----|-----|---------|
| 第一阶段 | 3个月 | 显示基础海图 | 长江航道武汉段 |
| 第二阶段 | 6-12个月 | 导航+预警功能 | 珠江流域、洞庭湖、太湖、东南沿海 |
| 第三阶段 | 1-3年 | 完整功能+全球 | 全球水域 |

## 二、系统架构需求

### 2.1 技术架构

#### 2.1.1 跨平台内核

| 项目 | 说明 |
|-----|-----|
| 核心技术 | C++核心逻辑 + Java API |
| 备选方案 | Qt/QML、.NET MAUI、Flutter、WebAssembly |
| 设计原则 | 一套内核多端复用，各端功能一致 |

**内核技术选型对比：**

| 技术方案 | 优点 | 缺点 | 适用场景 |
|---------|-----|-----|---------|
| C++核心 + Java API | 性能最优、跨平台复用 | 开发复杂度高 | 当前主选方案 |
| Qt/QML | 跨平台UI框架成熟 | 商业授权费用 | 桌面端备选 |
| .NET MAUI | 微软生态整合好 | Linux支持弱 | Windows为主场景 |
| Flutter | 移动端开发效率高 | 桌面端成熟度不足 | 移动端优先场景 |
| WebAssembly | Web端原生性能 | 桌面/移动端支持有限 | Web优先场景 |

**内部数据格式设计：**

| 项目 | 说明 |
|-----|-----|
| 存储格式 | SQLite数据库 + SpatiaLite扩展 |
| 几何存储 | 空间索引优化 |
| 坐标系 | WGS-84为主，支持转换 |
| 版本管理 | 数据版本号+增量更新 |

**数据API设计职责：**

| API模块 | 职责 |
|--------|-----|
| 数据导入API | S57/S101格式解析、内部格式转换 |
| 空间查询API | 范围查询、要素查询、空间关系判断 |
| 更新管理API | 增量更新、版本同步、冲突处理 |
| 导出接口API | 航线导出、报告生成、数据备份 |

#### 2.1.2 服务架构

| 阶段 | 架构模式 | 说明 |
|-----|---------|-----|
| 前期 | 端侧计算 | 完全本地计算，无需服务器 |
| 后期 | 云端协同 | 中心服务器处理数据、航线计算、预警分发 |

**架构演进路径：**

```
第一阶段（端侧计算）：
┌─────────────────────────────────────┐
│           客户端应用                │
│  ┌─────────┐ ┌─────────┐ ┌────────┐│
│  │ 渲染引擎 │ │ 航线计算 │ │预警引擎││
│  └─────────┘ └─────────┘ └────────┘│
│  ┌─────────────────────────────────┐│
│  │      本地数据存储(SQLite)       ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘

第二阶段（云端协同）：
┌──────────────┐     ┌──────────────┐
│   客户端应用  │◄───►│   云端服务   │
│  (轻量化)    │     │ ┌──────────┐ │
└──────────────┘     │ │航线计算   │ │
                     │ ├──────────┤ │
                     │ │预警分发   │ │
                     │ ├──────────┤ │
                     │ │数据同步   │ │
                     │ └──────────┘ │
                     └──────────────┘
```

## 三、数据需求

### 3.1 海图数据

#### 3.1.1 数据格式

| 格式 | 说明 | 优先级 |
|-----|-----|-------|
| S57 | IHO电子海图标准 | P0 |
| S101 | 新一代电子海图标准 | P0 |
| S102 | 高密度水深数据 | P1 |
| S111 | 表层海流数据 | P2 |
| S112 | 水位数据 | P2 |
| S124 | 航行通告 | P1 |

**S100系列数据详细说明：**

| 标准 | 全称 | 数据内容 | 应用场景 |
|-----|-----|---------|---------|
| S-100 | IHO通用水文数据模型 | 基础框架标准 | 所有S-100系列基础 |
| S-101 | 电子海图 | 航行障碍物、助航标志、水深等 | 替代S-57的新一代海图 |
| S-102 | 高密度水深 | 高分辨率海底地形 | 精细化水深显示 |
| S-111 | 表面流 | 海流流向流速 | 航行效率优化 |
| S-112 | 水位信息 | 水位预报数据 | 实时水深计算 |
| S-124 | 航行通告 | 航行警告、通知 | 安全预警 |
| S-129 | 富余水深管理 | UKC计算数据 | 智能航线规划 |

#### 3.1.2 数据处理

| 项目 | 说明 |
|-----|-----|
| 格式转换 | 转换为内部优化格式 |
| 空间索引 | R树或四叉树索引 |
| 数据压缩 | 支持数据压缩存储 |

**数据处理流程：**

```
原始数据(S57/S101) 
    ↓
解析验证(格式校验、完整性检查)
    ↓
坐标转换(统一至WGS-84)
    ↓
空间索引构建(R树/四叉树)
    ↓
内部格式存储(SQLite + SpatiaLite)
    ↓
渲染优化(LOD分级、样式预处理)
```

**要素显示阈值设计（遵循SCAMIN原则）：**

| 要素类型 | 缩放级别 | 显示策略 |
|---------|---------|---------|
| 水深点 | < 1:200,000 | 全部隐藏 |
| 水深点 | 1:100,000 - 1:200,000 | 稀疏显示（关键航道） |
| 水深点 | 1:50,000 - 1:100,000 | 常规显示（智能抽稀） |
| 水深点 | > 1:50,000 | 完整显示 |
| 助航标志 | < 1:50,000 | 仅显示灯塔、重要航标 |
| 助航标志 | > 1:50,000 | 显示所有航标 |
| 等深线 | 所有级别 | 根据比例尺调整密度 |

| 坐标系 | 说明 | 用途 |

**坐标系转换要求：**

| 项目 | 说明 |
|-----|-----|
| 转换精度 | 默认<0.1米 |
| 运行时切换 | 不允许 |
| 性能影响 | 投影数据性能上浮20% |
| 应用场景 | 宏观导航允许几十米误差，港口精密停泊要求厘米级 |

**坐标系转换参数：**

| 转换类型 | 转换参数 | 精度 |
|---------|---------|-----|
| WGS-84 ↔ CGCS2000 | 七参数转换 | 厘米级 |
| 北京54 → WGS-84 | 三参数/七参数 | 分米级 |
| 地方坐标 → WGS-84 | 需要当地参数 | 视参数而定 |

#### 4.3.5 多目标优化详述

**基本思路：**

航线规划需要同时考虑多个目标，这些目标之间往往存在冲突，需要在安全、经济、时效之间寻找帕累托最优解。

**考虑维度：**

| 维度 | 具体指标 | 约束条件 |
|------|---------|---------|
| 安全性 | UKC、与碍航物距离、航道宽度 | UKC≥安全阈值、距离≥安全距离 |
| 经济性 | 燃油消耗、航行距离 | 燃油消耗最小化 |
| 时效性 | 航行时间、ETA准确性 | 满足到港时间窗口 |
| 舒适性 | 转弯次数、加速度变化 | 减少剧烈操纵 |
| 环保性 | 碳排放、硫排放 | 符合排放控制区要求 |

**多目标优化流程：**

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 目标定义 | 确定优化目标及权重 |
| 2 | 约束建模 | 建立安全约束、时间约束等 |
| 3 | 算法选择 | 选择适合的多目标优化算法 |
| 4 | 候选解生成 | 生成多条帕累托最优航线 |
| 5 | 决策支持 | 提供决策支持，供驾驶员选择 |

**成熟方案：**

| 方案 | 算法 | 特点 | 适用场景 |
|------|------|------|---------|
| NSGA-II | 非支配排序遗传算法 | 收敛性好，分布均匀 | 通用多目标优化 |
| MOEA/D | 基于分解的多目标进化算法 | 计算效率高 | 大规模问题 |
| 粒子群算法 | 多目标粒子群优化 | 实现简单，收敛快 | 动态环境 |
| 两阶段算法 | 第一阶段全局+第二阶段局部 | 兼顾全局与局部 | 复杂约束问题 |

**待解决问题：**

| 问题 | 说明 | 研究方向 |
|------|------|---------|
| 动态环境适应 | 实时气象、交通变化 | 在线重规划、增量优化 |
| 高维目标 | 目标数量增加时计算复杂度 | 降维处理、目标聚合 |
| 约束处理 | 复杂约束条件下的可行解搜索 | 约束处理技术 |
| 实时性要求 | 需在短时间内给出优化结果 | 并行计算、近似算法 |

#### 4.3.6 分层规划策略

**全局粗规划+局部精调：**

这是一种典型的分层规划思想，可有效降低计算量：

| 层级 | 规划方法 | 目标 | 计算量 |
|------|----------|------|--------|
| 全局粗规划 | 改进A*、RRT*、网格搜索 | 生成可行路径骨架，避开大范围障碍物 | 低 |
| 局部精调 | DWA、TEB、MPPI | 考虑动态障碍物、水流、转向约束，平滑路径 | 中 |
| 融合优化 | 全局节点引导+局部滚动优化 | 兼顾全局最优性和局部动态适应性 | 适中 |

**算法框架：**

```
全局粗规划（离线）
    ↓
生成初始路径骨架
    ↓
局部精调（在线）
    ↓
考虑动态约束
    ↓
实时路径更新
```

**效果评估：**

| 指标 | 传统方法 | 分层规划 | 提升 |
|------|---------|---------|------|
| 规划时间 | 数分钟 | 数十秒 | 降低50%+ |
| 路径质量 | 局部最优 | 全局近优 | 提升10%+ |
| 动态适应性 | 差 | 好 | 显著提升 |

#### 4.3.8 潮汐时间窗耦合优化

**优化思路：**

将潮汐时间窗作为约束条件纳入航线规划模型，在满足通航水深要求的前提下，优化船舶通过关键航段的时间窗口。

**背景：**

| 项目 | 说明 |
|-----|-----|
| 问题来源 | 内河航道水深受潮汐影响，船舶需等待合适潮位才能通过 |
| 传统方式 | 船舶根据经验等待潮水，效率低下 |
| 优化目标 | 精准计算潮汐时间窗，减少等待时间 |

**优化模型：**

| 模型要素 | 说明 |
|---------|-----|
| 目标函数 | 最小化总航行时间（含等待时间） |
| 约束条件 | 潮汐时间窗约束、水深约束、航速约束 |
| 决策变量 | 各航段到达时间、航速、等待时间 |

**现状：**

| 方面 | 现状 |
|------|------|
| 研究进展 | 已有带潮汐时间窗的船舶路径规划模型（FSRPTTW） |
| 算法 | 列生成算法、混合整数规划 |
| 应用 | 支线船舶调度、港口泊位分配 |

**待解决问题：**

| 问题 | 说明 |
|------|------|
| 潮汐预报精度 | 潮汐预报存在误差，影响时间窗计算 |
| 动态水位变化 | 实际水位与预报不一致 |
| 多船协同 | 多船竞争同一时间窗 |
| 不确定性处理 | 天气、交通等不确定因素 |

**可达成目标：**

| 目标 | 预期效果 |
|------|---------|
| 减少等待时间 | 优化后等待时间减少30-50% |
| 提高准点率 | ETA准确率提升至90%+ |
| 降低燃油消耗 | 通过优化航速降低燃油消耗5-10% |
| 提升通航效率 | 航道通航能力提升10-20% |

#### 4.3.11 手动规划耗时原因分析

**为何近海及港口航线仍以手动规划为主？**

| 原因类型 | 具体表现 |
|---------|---------|
| 环境复杂性 | 动态水深与潮汐实时变化、复杂碍航物、交通密集 |
| 法规要求 | SOLAS公约要求航线计划必须由持证驾驶员完成，并经过船长审批 |
| 安全冗余 | 手动规划允许人在环中反复检查，对潜在风险保留主观判断 |
| 技术成熟度 | 实时AIS、气象、水文、港口调度数据标准不一，实时融合困难 |

**缩短规划耗时的市场需求：**

| 需求 | 说明 |
|-----|-----|
| 提高运营效率 | 船舶周转率提升需要快速响应临时货单、港口窗口期变更 |
| 降低人力成本 | 高级船员工作负荷重，自动规划可释放其精力 |
| 应对复杂水域 | 近海交通密度加大，人工规划易遗漏风险 |
| 绿色航运需求 | 优化航线以降低油耗，需要综合考虑气象、流场等多变量 |

**缩短耗时的可行方法：**

| 方法 | 说明 |
|-----|-----|
| 数据驱动的高精度动态建模 | 整合AIS、潮汐预报、气象数值预报、港口生产系统，建立统一时空数据库 |
| 智能路径规划算法 | 采用进化算法、粒子群算法在安全、经济、时效等多目标间求帕累托最优解 |
| 人机协同的半自动规划 | 系统自动生成3-5条推荐航线，驾驶员快速选择并微调 |
| 标准化与数字化 | 将各港口航行指南、引航方案转为机器可读的规则库 |

#### 10.5.5 数字化留痕方案

**三层机制设计：**

| 层级 | 机制 | 实现方式 |

**实现路径：**

| 步骤 | 操作 | 说明 |

| 5 | 同步上传 | 审批后自动同步至岸基系统 |
| 6 | 归档存储 | 按航次归档，支持历史查询 |

**与Excel方式对比：**

| 对比项 | Excel方式 | 数字化系统 |
|-------|----------|-----------|

| 历史追溯 | 困难 | 一键查询 |
| 防篡改 | 无 | 区块链/哈希链 |

## 十三、UKC计算详述

### 13.1 计算公式

```
UKC = H - T - S - E
```

| 参数 | 说明 |
|-----|-----|
| H | 实际水深（海图水深 + 潮高） |
| T | 船舶吃水 |
| S | 航行下沉量（Squat） |
| E | 测量误差和安全余量 |

### 13.2 下沉量计算

**Barrass公式：**
```
Squat = Cb × V² / 100 × K
```

| 参数 | 说明 |
|-----|-----|
| Cb | 方形系数 |
| V | 航速（节） |
| K | 受限系数 |

### 13.3 UKC标准

| 航道类型 | 最小UKC要求 |
|---------|------------|
| 外海航道 | 吃水的20% |
| 港口航道 | 吃水的10% |
| 内河航道 | 吃水的10-15% |
| 受限水域 | 吃水的15-20% |

### 16.2 吴淞高程转换

#### 16.2.1 基准面关系

| 基准类型 | 定义 | 应用场景 |
|---------|-----|---------|
| 吴淞高程 | 上海吴淞口验潮站1871-1900年最低潮位为零点 | 长江航道水位 |
| 1985国家高程基准 | 青岛验潮站1952-1979年平均海面 | 全国统一高程 |
| 理论深度基准面 | 理论最低潮面 | 海图水深基准 |

#### 16.2.2 转换公式

**吴淞高程→1985国家高程基准：**
```
H_1985 = H_吴淞 - 1.629米
```
*注：此值为长江口区域常用近似值，不同航段存在差异*

**1985国家高程基准→理论深度基准面：**
```
H_深度基准 = H_1985 + ζ - MDT
```
| 参数 | 说明 |
|-----|-----|
| ζ | 高程异常（似大地水准面与椭球面之差） |
| MDT | 平均海面地形 |

#### 16.2.3 转换精度

| 区域 | 转换精度 |
|-----|---------|
| 陆域 | 优于2.5厘米 |
| 海域 | 优于4.1厘米 |
| 全域 | 优于3厘米 |

### 16.3 桥梁净高传感器方案

#### 16.3.1 查询渠道

| 查询渠道 | 代表系统 | 覆盖范围 | 特点 |
|---------|---------|---------|------|
| 手机APP | 粤航通App | 广东230余座桥梁 | 一站式查询、过闸申报、航线规划 |
| 微信小程序 | 靖江通航净高查询 | 跨航道桥梁 | 雷达水位计+主动预警 |
| 航道图系统 | 长江电子航道图 | 长江干线 | 集成在电子航道图中 |
| 现场显示 | LED大屏 | 桥梁现场 | 实时净高数值 |

#### 16.3.2 查询内容

| 信息类别 | 具体内容 | 武汉示例 |
|---------|---------|---------|
| 基础信息 | 桥梁名称、位置、设计净高 | 武汉长江大桥：设计净高18米 |
| 实时净高 | 当前水位下的实际净空 | 根据实时水位动态计算 |
| 水位数据 | 当前水位、历史水位、潮汐预报 | 汉口站水位、未来24小时预报 |
| 安全提示 | 限高船舶预警、通航状态 | 超高船舶预警 |

#### 16.3.3 数据组织结构

桥梁净高数据的组织采用"基础静态数据+动态实时数据"的双层结构：

```
桥梁净高数据
├── 静态基础数据（S-125标准）
│   ├── 桥梁ID（唯一标识）
│   ├── 桥梁名称、位置（经纬度）
│   ├── 设计净高、通航孔参数
│   ├── 梁底高程（绝对高程）
│   └── 相关航标信息
└── 动态实时数据
    ├── 实时水位（水位计采集）
    ├── 实时净高 = 梁底高程 - 实时水位
    ├── 水位变化趋势
    └── 预警状态（正常/注意/危险）
```

#### 16.3.4 武汉跨江大桥案例

武汉拥有11座跨长江大桥，每座桥梁的净高查询存在不同问题：

| 桥梁名称 | 设计净高 | 主要问题 | 实际影响 |
|---------|---------|---------|---------|
| 武汉长江大桥 | 18m | 建成最早，净高最低 | 洪水期大型船舶无法通过 |
| 武汉长江二桥 | 24m | 位于弯道，通航条件复杂 | 需结合弯道水流考虑 |
| 鹦鹉洲长江大桥 | 30m | 悬索桥，净高受温度影响 | 钢索垂度随温度变化 |
| 杨泗港长江大桥 | 30m | 双层桥梁，结构复杂 | 需考虑下层桥梁影响 |
| 天兴洲长江大桥 | 24m | 公铁两用，通航孔多 | 不同通航孔净高不同 |

**常见查询问题：**

| 问题类型 | 具体表现 | 影响 |
|---------|---------|-----|
| 数据更新滞后 | 水位数据每15-30分钟更新一次，洪水期水位暴涨时无法实时反映 | 净高信息不准 |
| 精度不足 | 部分桥梁净高计算误差达10-20cm | 对空载大型船舶存在安全隐患 |
| 坐标系不一致 | 水位基准面与桥梁高程基准面不统一 | 查询结果需人工换算 |
| 缺少预测功能 | 只能查当前净高，无法预测未来几小时的净高变化趋势 | 无法提前规划 |
| 信息孤岛 | 不同桥梁数据分散在不同系统，需逐一查询 | 查询效率低 |

#### 16.3.5 净高预警基本思路

**预警基本流程：**

```
水位传感器实时采集 
    ↓
数据融合与净高计算（梁底高程-实时水位）
    ↓
与船舶高度+安全余量比较
    ↓
触发条件判断 → 超出阈值 → 预警发布
    ↓
预警方式：APP推送/VHF广播/现场LED/短信
```

**预警级别设置：**

| 预警级别 | 触发条件 | 响应措施 |
|---------|---------|---------|
| 黄色预警 | 净高 < 船舶高度+2m | 提醒注意，建议减速观察 |
| 橙色预警 | 净高 < 船舶高度+1m | 建议等待潮水或绕航 |
| 红色预警 | 净高 < 船舶高度+0.5m | 禁止通过，强制等待 |

**预警难点：**

| 难点 | 说明 | 解决思路 |
|-----|-----|---------|
| 数据实时性 | 水位变化快，数据更新滞后 | 提高采集频率，引入预测算法 |
| 精度要求 | 船舶高度近18米，误差10cm即有风险 | 采用高精度传感器，多源数据融合 |
| 坐标转换 | 不同基准面需换算 | 建立统一转换模型 |
| 船舶高度获取 | 需知道船舶实际高度 | 建立船舶数据库，结合AIS信息 |
| 预测能力 | 需预测未来净高 | 结合潮汐预报和水文模型 |
