# Multi-Role Reviewer 技能使用指南

多角色交叉评审技能，支持多个专家角色独立评审、交叉投票、权重计算和迭代改进。

---

## 快速开始

### 基本用法

```
> 请对xxx.md进行多角色交叉评审，直到评分>95
```
```
请对model_design.md进行多角色交叉评审，直到评分>95,要求至少要有5个相关角色，其中一个角色为GIS开源领域的核心架构师与关键代表人物
```
### 指定角色

```
> 请以数据库架构师、性能工程师、安全专家三个角色交叉评审xxx.md
```

### 系统行为

1. 自动选择专家角色（或使用指定角色）
2. 每个角色独立评审
3. 角色之间互相投票
4. 按权重改进文档
5. 评分 ≥ 95 时停止

```
4. 新权重：(1.0 + 1.0 + 1.0) / 2 = 1.0（高优先级）
```

## 核心概念

### 与 technical-reviewer 的区别

| 特性 | technical-reviewer | multi-role-reviewer |
|------|-------------------|---------------------|
| **专家角色** | 单个专家 | 多个专家（2-5个） |
| **评审方式** | 顺序（修复→重新评审） | 并行+竞争 |
| **决策机制** | 专家决定 | 角色投票决定正确性 |
| **速度** | 较快（2-5分钟） | 较慢（5-15分钟） |
| **适用场景** | 单领域深度评审 | 多视角综合评审 |

### 角色权重 vs 发现权重

| 概念 | 定义 | 用途 |
|------|------|------|
| **角色权重** | 基于专家发现准确率分配的权重 | 用于最终评分计算 |
| **发现权重** | 基于投票结果分配的权重 | 用于优先级排序 |

## 评审流程

```
Phase 1: 角色选择与初始评审
├── 根据文档领域选择2-5个专家角色
├── 每个角色独立评审文档
└── 输出：问题、建议、维度评分

Phase 2: 交叉评审与投票
├── 角色互相评审发现
├── 投票：✅ 同意 / ⚠️ 部分同意 / ❌ 不同意
└── 计算权重：权重 = 同意票数 / 总票数

Phase 3: 整合与优先级排序
├── 高权重发现（>0.7）：必须解决
├── 中权重发现（0.4-0.7）：应该解决
└── 低权重发现（<0.4）：可选

Phase 4: 文档改进
├── 按优先级解决问题
├── 更新文档版本
└── 记录变更

Phase 5: 重新评估
├── 计算新评分
├── 评分 ≥ 95：完成
└── 评分 < 95：返回Phase 1

Phase 6: 最终报告
├── 生成综合评审报告
├── 包含所有角色视角和投票结果
└── 跟踪迭代历史
```

## 投票规则

### 投票类型

| 投票 | 含义 | 对发现权重的影响 |
|------|------|------------------|
| ✅ 同意 | 问题有效，建议正确 | +1 到同意计数 |
| ⚠️ 部分同意 | 问题存在但建议需调整 | +0.5 到同意计数 |
| ❌ 不同意 | 问题无效或建议错误 | +0 到同意计数 |

### 权重计算

```
发现权重 = Σ(投票值) / 投票角色数

示例：
发现："DbConnection接口过大"
投票：数据库架构师(✅)、性能工程师(✅)、安全专家(⚠️)
权重 = (1.0 + 1.0 + 0.5) / 3 = 0.83 → 高优先级
```

## 评分标准

| 分数范围 | 等级 | 说明 |
|----------|------|------|
| 95-100 | A+ (优秀) | 可直接用于生产环境 |
| 90-94 | A (优秀) | 小问题，可用于生产环境 |
| 85-89 | B+ (良好) | 设计合理，需小幅修改 |
| 80-84 | B (良好) | 基本设计合理，需修改 |
| 70-79 | C (及格) | 存在明显问题，需大幅修改 |
| <70 | D (不及格) | 存在严重问题，需重新设计 |

## 发现状态定义

| 状态 | 标识 | 说明 |
|------|------|------|
| ✅ 已解决 | RESOLVED | 问题已修复，已验证通过 |
| ⏳ 进行中 | IN_PROGRESS | 问题正在处理中 |
| 📋 待处理 | PENDING | 问题已确认，等待处理 |
| ⏭️ 后续版本 | DEFERRED | 问题已确认，推迟到后续版本 |
| ❌ 不采纳 | REJECTED | 问题无效或建议不适用 |

## 触发关键词

当用户请求包含以下关键词时，系统会自动调用此技能：

- "交叉评审"、"多角色评审"、"cross-review"
- "多角色"、"投票评审"、"多角度评审"
- "交叉审核"

**注意**：如果只包含"评审"、"审计"等关键词（无交叉/多角色关键词），系统会调用 `technical-reviewer` 技能。

## 时间与资源估算

### 时间估算

| 阶段 | 预估时间 |
|------|----------|
| Phase 1: 角色选择 | 10-30秒 |
| Phase 2: 独立评审 | 每个角色1-3分钟 |
| Phase 3: 交叉投票 | 30-60秒 |
| Phase 4: 文档改进 | 1-5分钟 |
| Phase 5: 重新评估 | 30-60秒 |
| **每次迭代总计** | **3-10分钟** |

### Token估算

```
总Token ≈ 3,000 + (角色数 × 1,500) + (文档页数 × 300) + (发现数 × 800) + (迭代次数 × 5,000)

示例（3角色，10页文档，15个发现，2次迭代）:
= 3,000 + (3 × 1,500) + (10 × 300) + (15 × 800) + (2 × 5,000)
= 32,500 tokens
```

## 常见问题

### Q1: 如何指定多个角色？

在请求中列出角色：
```
请以[角色1]、[角色2]、[角色3]三个角色交叉评审xxx文档
```

### Q2: 评审报告保存在哪里？

默认保存在目标文档同目录下，命名为 `<文档名>_multi_audit.md`。

### Q3: 最多迭代多少次？

最多5次迭代。如果5次迭代后仍未达到95分，会生成失败报告并提供用户选项。

### Q4: 角色之间有分歧怎么办？

系统会：
1. 计算发现权重
2. 请求分歧角色解释理由
3. 重新投票
4. 如仍有分歧，使用多数投票或提交用户决定

### Q5: 如何处理无法解决的问题？

系统会提供以下选项：
- A. 接受当前评分
- B. 手动干预解决阻塞问题
- C. 缩小范围（仅关注高权重问题）
- D. 添加更多专家角色
