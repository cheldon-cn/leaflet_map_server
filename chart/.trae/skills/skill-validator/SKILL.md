# Skill Validator

**Skill质量评估工具** - 从多个维度全面评估skill质量，生成详细的评估报告，帮助提升skill质量。

---

## 一、Skill概述

### 1.1 功能定位

本Skill用于对其他Skill进行全面质量评估，确保：

1. **多维度评估**: 从功能、文档、可用性、可靠性、性能、可维护性六个维度评估
2. **科学评分**: 基于权重模型的科学评分体系
3. **问题追踪**: 详细记录问题、原因、修改方法、修改前后变化
4. **质量提升**: 提供具体的改进建议，真正提高skill质量

### 1.2 触发条件

当出现以下情况时，应调用本Skill：

- 用户需要对skill进行质量评估
- 用户准备发布skill但不确定质量是否达标
- 用户询问skill质量、评估、验证等关键词
- 用户提到"skill评估"、"质量检查"、"验证skill"等

### 1.3 核心约束

**❌ 不要做的事**：
- 不要跳过任何维度的评估
- 不要忽略发现的问题
- 不要给出无依据的评分
- 不要遗漏问题的修改建议
- 不要在评估报告中使用模糊表述

### 1.4 与其他Skill的关系

```
┌─────────────────────────────────────────────────────────────┐
│                    Skill协作关系                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [待评估Skill] → skill-validator → [评估报告]                │
│                         ↓                                   │
│                  [问题列表]                                  │
│                         ↓                                   │
│                  skill-creator → [改进后的Skill]             │
│                         ↓                                   │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、评估维度详解

### 2.1 功能维度 (权重: 25%)

#### 2.1.1 功能完整性 (15%)

| 指标 | 说明 | 评分标准 |
|------|------|----------|
| 需求覆盖度 | 是否完整实现描述中的功能 | 100%覆盖=10分, 80%=8分, 60%=6分, <60%=不及格 |
| 边界场景处理 | 是否处理边界情况和异常场景 | 完整处理=10分, 部分=6分, 无=0分 |
| 核心路径完整性 | 主要使用流程是否完整 | 完整=10分, 基本完整=7分, 不完整=3分 |

#### 2.1.2 功能正确性 (10%)

| 指标 | 说明 | 评分标准 |
|------|------|----------|
| 输出准确性 | 输出结果是否正确 | 完全正确=10分, 基本正确=7分, 有错误=3分 |
| 逻辑一致性 | 内部逻辑是否自洽 | 自洽=10分, 基本自洽=6分, 有矛盾=2分 |
| 符合规范 | 是否遵循相关标准/规范 | 完全符合=10分, 基本符合=6分, 不符合=0分 |

---

#### 2.2.1 文档结构 (12%)

| 示例丰富 | 有典型使用示例 | 丰富=10分, 有示例=6分, 无示例=0分 |
| 约束明确 | 明确说明"不要做的事" | 明确=10分, 有说明=5分, 无=0分 |

#### 2.2.2 文档可读性 (8%)

| 格式规范 | 表格、代码块、图表使用恰当 | 规范=10分, 基本规范=6分, 不规范=2分 |

## 三、评分体系

### 3.1 综合评分公式

```
质量得分 = Σ(维度权重 × 维度得分)

维度权重：
- 功能维度: 25%
- 文档维度: 20%
- 可用性维度: 20%
- 可靠性维度: 15%
- 性能维度: 10%
- 可维护性维度: 10%

维度得分 = Σ(指标得分 × 指标权重) / Σ指标权重
```

### 3.2 质量等级划分

| 等级 | 得分范围 | 说明 | 发布建议 |
|------|----------|------|----------|
| 🟢 S级（卓越） | 95-100 | 各维度表现卓越，可作为标杆 | 强烈推荐发布 |
| 🟢 A级（优秀） | 85-94 | 各维度表现优秀，可直接使用 | 推荐发布 |
| 🟡 B级（良好） | 75-84 | 整体良好，有少量改进空间 | 建议改进后发布 |
| 🟠 C级（合格） | 60-74 | 基本可用，需要改进 | 需要改进后发布 |
| 🔴 D级（不合格） | <60 | 存在严重问题 | 需要重大改进或重写 |

### 3.3 维度等级判定

| 维度得分 | 等级 | 说明 |
|----------|------|------|
| 90-100 | 🟢 优秀 | 该维度表现优秀 |
| 75-89 | 🟡 良好 | 该维度表现良好 |
| 60-74 | 🟠 合格 | 该维度基本合格 |
| <60 | 🔴 不合格 | 该维度需要改进 |

## Step 7: 迭代修改机制

### 7.1 迭代规则

**核心规则**：

| 条件 | 操作 |
|------|------|
| 综合得分 < 96分 | 自动进入迭代修改流程，根据评估报告改进Skill |
| 综合得分 ≥ 96分 | 输出提示："🎉 Skill很优秀，可以直接使用了！" |

**迭代限制**：

| 限制项 | 值 | 说明 |
|--------|-----|------|
| 单轮对话最大迭代次数 | 6次 | 超过6次仍未达标，输出当前结果并建议人工介入 |
| 迭代触发条件 | 得分 < 96 | 每次评估后自动判断 |
| 迭代终止条件 | 得分 ≥ 96 或 达到最大迭代次数 | 满足任一条件即终止 |

### 7.3 迭代修改执行步骤

每次迭代执行以下步骤：

#### Step 7.3.1: 分析问题优先级

```
1. 从评估报告中提取所有问题
2. 按优先级排序：P0 > P1 > P2 > P3
3. 确定本次迭代要解决的问题（优先处理P0和P1）
```

#### Step 7.3.2: 执行修改

```
1. 针对每个问题，按照评估报告中的"修改建议"执行修改
2. 更新SKILL.md文件
3. 记录修改内容
```

#### Step 7.3.3: 更新评估报告

```
1. 重新评估修改后的Skill
2. 更新评估报告文件：[skill-name]-skill-validator-result.md
3. 在报告中添加迭代记录
```

### 7.6 迭代修改约束

**❌ 迭代过程中不要做的事**：

1. **不要跳过问题**: 必须按优先级处理所有发现的问题
2. **不要过度修改**: 每次迭代只修改评估报告中指出的问题
3. **不要忽略得分变化**: 每次迭代后必须重新评估并记录得分变化
4. **不要无限迭代**: 严格遵守6次迭代上限
5. **不要删除已有内容**: 修改时保留Skill的核心结构和功能

**✅ 迭代过程中要做的事**：

1. **按优先级修改**: 先处理P0、P1问题，再处理P2、P3问题
2. **记录每次修改**: 详细记录修改内容和原因
3. **更新评估报告**: 每次迭代后更新评估报告
4. **对比得分变化**: 展示修改前后的得分对比
5. **标记问题状态**: 更新问题的处理状态

## 六、性能说明

### 6.1 评估复杂度

| Skill规模 | 文件大小 | 预期评估时间 | 说明 |
|-----------|----------|--------------|------|
| 小型 | < 20KB | < 10秒 | 基本评估，快速完成 |
| 中型 | 20-50KB | 10-30秒 | 完整评估，包含详细分析 |
| 大型 | 50-100KB | 30-60秒 | 详细评估，建议关注关键问题 |
| 超大型 | > 100KB | > 60秒 | 建议分段评估或聚焦核心问题 |

### 6.2 性能影响因素

| 因素 | 影响程度 | 说明 |
|------|----------|------|
| 文档长度 | 高 | 文档越长，解析和评估时间越长 |
| 章节数量 | 中 | 章节越多，结构分析越复杂 |
| 示例数量 | 低 | 示例主要影响文档维度评估 |
| 迭代次数 | 高 | 每次迭代需要重新评估 |

### 6.3 性能优化建议

#### 6.3.1 评估前优化

1. **聚焦核心问题**: 对于大型Skill，可先关注P0/P1问题
2. **分段评估**: 将大型Skill拆分为多个部分分别评估
3. **预检查**: 先进行快速预检查，识别主要问题区域

#### 6.3.2 迭代优化

1. **批量修改**: 每次迭代集中修改多个问题，减少迭代次数
2. **优先级排序**: 先处理高优先级问题，可能连带解决低优先级问题
3. **增量评估**: 对于小改动，可只重新评估受影响的维度

#### 6.3.3 大型Skill评估策略

```
┌─────────────────────────────────────────────────────────────┐
│              大型Skill评估策略                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 快速预评估                                         │
│  ├── 识别主要问题区域                                       │
```
