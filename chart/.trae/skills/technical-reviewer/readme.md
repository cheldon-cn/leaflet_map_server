# Technical Reviewer 技能使用指南

技术设计文档评审技能，支持专家角色扮演、结构化输出和迭代改进。

---

## 快速开始

### 基本用法

```
> 请评审xxx.md，直到评分>95
```

### 指定专家角色

```
> 请以数据库架构师的身份评审database_design.md
```

### 系统行为

1. 自动检测专家角色（基于文档类型）
2. 从专家视角评审文档
3. 创建评审报告并打分
4. 若评分 < 95，迭代改进文档
5. 评分 ≥ 95 时停止

---

## 使用示例

### 示例1：自动检测专家角色

**用户输入：**
```
请评审database_model_design.md
```

**系统行为：**
1. 自动检测：Database Architect（从文件名"database"推断）
2. 评审6个维度：架构合理性、扩展性、性能保证、生态兼容性、落地可行性、文档完整性
3. 发现问题：连接池缺少耗尽处理、批量插入缺少错误恢复
4. 创建评审报告：`database_model_design_audit.md`
5. 评分：82分，需要改进
6. 改进文档，重新评审
7. 迭代直到评分>95

## 专家角色定义

### 自动检测规则

| 关键词模式 | 检测角色 | 关注领域 |
|-----------|----------|----------|
| `database`, `db_`, `storage`, `sql` | Database Architect | 存储引擎、索引、MVCC、吞吐量 |
| `layer`, `model`, `geometry`, `spatial` | GIS Core Developer | OGC标准、空间操作、内存管理 |
| `api`, `interface`, `sdk` | API Architect | 接口设计、版本管理、兼容性 |
| `performance`, `benchmark`, `optimize` | Performance Engineer | 热点路径、优化、基准测试 |
| `security`, `auth`, `encrypt` | Security Expert | 认证、加密、漏洞防护 |
| `network`, `protocol`, `rpc` | Network Architect | 协议设计、延迟、可靠性 |
| `ui`, `ux`, `frontend` | Frontend Architect | 组件设计、状态管理、用户体验 |
| `test`, `qa`, `quality` | QA Architect | 测试策略、覆盖率、自动化 |

### 角色选择模式

| 模式 | 工作方式 | 使用场景 |
|------|----------|----------|
| **自动检测** | 基于文档内容/文件名自动匹配 | 标准评审、快速评估 |
| **用户指定** | 用户在请求中明确指定角色 | 领域深度评审 |
| **混合模式** | 自动检测 + 用户确认 | 最灵活，推荐默认 |

## 评审维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 架构合理性 | 20% | 设计是否合理、层次是否清晰 |
| 扩展性 | 15% | 是否易于扩展、支持新场景 |
| 性能保证 | 20% | 性能考量是否充分 |
| 生态兼容性 | 15% | 与现有生态的兼容程度 |
| 落地可行性 | 15% | 实现难度、工作量评估 |
| 文档完整性 | 15% | 文档是否详尽、示例是否丰富 |

## 评分标准

| 分数范围 | 等级 | 说明 |
|----------|------|------|
| 95-100 | A+ (优秀) | 可直接用于生产环境 |
| 90-94 | A (优秀) | 小问题，可用于生产环境 |
| 85-89 | B+ (良好) | 设计合理，需小幅修改 |
| 80-84 | B (良好) | 基本设计合理，需修改 |
| 70-79 | C (及格) | 存在明显问题，需大幅修改 |
| <70 | D (不及格) | 存在严重问题，需重新设计 |

## 问题风险等级

| 等级 | 符号 | 定义 | 处理要求 |
|------|------|------|----------|
| 高 | 🔴 | 影响稳定性/数据一致性 | 必须在当前版本修复 |
| 中 | 🟠 | 影响性能/用户体验 | 应在当前版本修复 |
| 低 | 🟡 | 影响开发效率 | 可在后续版本修复 |

## 问题状态跟踪

### 状态生命周期

```
📋 待解决 ──► 🔄 进行中 ──► ✅ 已解决(vX.X)
```

| 状态 | 含义 |
|------|------|
| 📋 待解决 | 问题已识别，待处理 |
| 🔄 进行中 | 正在处理中 |
| ✅ 已解决(vX.X) | 已在版本X.X中解决 |

### 状态更新流程

```
Step 1: 识别问题 → 添加到报告，状态: 📋 待解决
Step 2: 修改目标文档 → 更新版本号
Step 3: 更新评审报告状态 → 📋 待解决 → ✅ 已解决(vX.X)
Step 4: 重新评审 → 计算新得分
Step 5: 更新文件头最终得分 → 迭代直到达标
```

## 评审报告结构

评审报告包含以下章节：

| 章节 | 内容 |
|------|------|
| 一、评审概述 | 评审方法说明 |
| 二、优势分析 | 架构/性能/工程优势 |
| 三、风险与问题 | 高/中/低风险问题 |
| 四、扩展性分析 | 支持和待扩展场景 |
| 五、生态兼容性 | 与现有库的兼容性 |
| 六、落地可行性 | 技术可行性、工作量评估 |
| 七、改进建议 | 高/中/低优先级建议 |
| 八、评分 | 维度得分 + 综合得分 |
| 九、评审签名 | 评审者、日期、状态 |
| 十、版本跟踪 | 修订历史、建议状态 |

## 与 multi-role-reviewer 对比

| 特性 | technical-reviewer | multi-role-reviewer |
|------|-------------------|---------------------|
| **专家角色** | 单个专家（自动检测或用户指定） | 多个专家（2-5个角色） |
| **评审深度** | 单视角深度分析 | 多视角广度覆盖 |
| **问题验证** | 单专家判断 | 交叉投票权重计算 |
| **速度** | 较快（2-5分钟） | 较慢（5-15分钟） |
| **Token消耗** | 较低（~15,000） | 较高（~30,000+） |
| **适用场景** | 单领域设计文档 | 跨领域综合评审 |

### 选择建议

| 文档类型 | 推荐技能 | 原因 |
|----------|----------|------|
| 单领域设计（数据库、API、UI） | technical-reviewer | 深度领域专业知识 |
| 多领域设计（系统架构） | multi-role-reviewer | 需要多视角评审 |
| 快速检查 | technical-reviewer | 更快完成 |
| 关键生产文档 | multi-role-reviewer | 更全面的验证 |

## 时间与资源估算

### 时间估算

| 步骤 | 预估时间 |
|------|----------|
| 读取文档 | 10-30秒 |
| 专家评审 | 1-3分钟 |
| 创建评审报告 | 30-60秒 |
| 迭代改进 | 每次迭代1-5分钟 |
| **总计（无迭代）** | **2-5分钟** |
| **总计（含迭代）** | **5-20分钟** |

### Token估算

```
总Token ≈ 2,000 + (文档页数 × 300) + (迭代次数 × 5,000)

示例（10页文档，2次迭代）:
= 2,000 + (10 × 300) + (2 × 5,000)
= 15,000 tokens
```

当用户请求包含以下关键词时，系统会自动调用此技能：

- "评审"、"审计"、"review"、"audit"

## 常见问题

### Q1: 如何指定专家角色？

在请求中明确指定：
```
请以[专家角色]的身份评审xxx文档
```

### Q2: 评审报告保存在哪里？

默认保存在目标文档同目录下，命名为 `<文档名>_audit.md`。

### Q3: 最多迭代多少次？

最多5次迭代。如果5次迭代后仍未达到95分，会生成失败报告并提供用户选项。

### Q4: 如何处理无法解决的问题？

系统会提供以下选项：
- A. 接受当前评分
- B. 手动干预解决阻塞问题
- C. 缩小范围（仅关注高/中风险问题）
- D. 请求多角色评审
