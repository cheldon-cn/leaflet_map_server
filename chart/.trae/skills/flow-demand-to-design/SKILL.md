# Flow: Demand to Design

端到端需求到设计转换流程Skill，自动编排需求分析、设计生成、验证评审的完整工作流，确保产出高质量设计文档。

---

## 一、Skill概述

### 1.1 功能定位

本Skill用于编排从原始需求文档到高质量设计文档的完整转换流程，确保：

1. **流程自动化**：自动串联5个步骤，无需人工干预
2. **质量保证**：每步输出验证，迭代改进直到达标
3. **追溯完整**：建立需求到设计的完整追溯链
4. **风险控制**：自动识别并处理流程中的问题

### 1.2 Quick Start (5分钟)

**基本用法:**
```
> 将需求文档转换为设计文档
```

**指定输出目录:**
```
> 转换 yangzt_demand.md 为设计文档
```

**That's it!** 系统会自动执行：
1. 需求分析 → 需求规格说明书
2. 设计生成 → 设计文档初稿
3. 设计验证 → 验证报告
4. 设计评审 → 评审报告
5. 迭代改进 → 直到评分≥95

**快速输出示例:**
```markdown
## 流程执行报告

### Step 1: 需求分析 ✅
- 输入: yangzt_demand.md
- 输出: yangzt_requirement_spec.md
- 需求数: 12个功能需求, 5个非功能需求

### Step 2: 设计生成 ✅
- 输入: yangzt_requirement_spec.md
- 输出: yangzt_design.md
- 模块数: 4个核心模块

### Step 3: 设计验证 ✅
- 输入: yangzt_design.md
- 输出: 验证报告
- 结果: 通过所有检查项

### Step 4: 设计评审 ✅
- 输入: 验证后的设计文档
- 输出: 评审报告
- 评分: 96/100

### Step 5: 迭代改进 ✅
- 迭代次数: 1次
- 最终评分: 96/100
- 状态: 达标

### 最终产物
- yangzt_requirement_spec.md
- yangzt_design.md
- yangzt_design_audit.md
```

### 1.3 性能预期

| 需求规模 | 步骤数 | 处理时间 | 迭代次数 |
|----------|--------|----------|----------|
| 小型 (<10条) | 5 | 3-5分钟 | 0-1次 |
| 中型 (10-30条) | 5 | 5-10分钟 | 1-2次 |
| 大型 (30-50条) | 5 | 10-20分钟 | 2-3次 |
| 超大型 (>50条) | 5 | 20-40分钟 | 3-5次 |

**性能影响因素:**
- 需求复杂度: 复杂需求需要更多迭代
- 设计质量: 初始设计质量差会增加迭代次数
- 评审标准: 严格的评审标准需要更多改进

### 1.4 触发条件

当出现以下情况时，应调用本Skill：

- 用户需要将需求文档转换为设计文档
- 用户提到"需求转设计"、"需求到设计"、"生成设计"
- 用户要求"完整流程"、"端到端转换"
- 用户有原始需求文档，需要产出设计文档
- 用户不确定如何从需求得到设计

### 1.5 核心约束（必读）

**✅ 必须做的事**：
- 按顺序执行5个步骤
- 每步输出必须验证通过
- 评分<95时自动迭代改进
- 保留所有中间产物
- 建立完整的追溯矩阵

**❌ 不要做的事**：
- 不要跳过任何步骤
- 不要在验证失败时继续
- 不要忽略评审意见
- 不要删除中间产物
- 不要在评分未达标时结束

### 2.2 Step 1: 需求分析

**调用Skill**: demand-analyzer

**输入**:
- 原始需求文档：`[industry]_demand.md`
- 格式：Markdown或纯文本

**输出**:
- 需求规格说明书：`[industry]_requirement_spec.md`
- 需求列表（功能需求、非功能需求）
- 优先级矩阵（MoSCoW分类）
- 风险清单
- 需求追溯矩阵

**验证规则**:
| 验证项 | 通过条件 |
|--------|----------|
| 需求数量 | 功能需求≥1个 |
| 验收标准 | 每个需求都有验收标准 |
| 优先级分配 | 所有需求已分配优先级 |
| 冲突检测 | 无未解决的冲突 |

**失败处理**:
- 返回需求分析报告
- 标记问题需求
- 提供补充建议

### 2.3 Step 2: 设计生成

**调用Skill**: design-generator

**输入**:
- 需求规格说明书：`[industry]_requirement_spec.md`

**输出**:
- 设计文档：`[industry]_design.md`
- 架构图
- 模块列表
- 接口定义
- 需求-设计追溯矩阵

**验证规则**:
| 验证项 | 通过条件 |
|--------|----------|
| 需求覆盖 | 100%功能需求有设计对应 |
| 模块定义 | 每个模块职责清晰 |
| 接口定义 | 每个接口有明确输入输出 |
| 技术选型 | 技术选型有理由 |

**失败处理**:
- 返回部分设计结果
- 标记需要决策的点
- 提供多个备选方案

### 2.4 Step 3: 设计验证

**调用Skill**: design-doc-validator

**输入**:
- 设计文档：`[industry]_design.md`

**输出**:
- 验证报告
- 改进建议
- 问题清单

**检查项**:
| 检查项 | 说明 |
|--------|------|
| 结构完整性 | 设计文档包含所有必要章节 |
| 需求覆盖度 | 所有需求都有设计对应 |
| 一致性 | 设计内部无矛盾 |
| 可实现性 | 设计可被实现 |

**失败处理**:
- 返回验证报告
- 列出所有问题
- 提供修复建议

### 2.5 Step 4: 设计评审

**调用Skill**: multi-role-reviewer

**输入**:
- 验证后的设计文档

**输出**:
- 评审报告
- 评分（0-100）
- 改进建议

**评审维度**:
| 维度 | 权重 | 说明 |
|------|------|------|
| 架构合理性 | 25% | 架构设计是否合理 |
| 扩展性 | 20% | 是否易于扩展 |
| 性能保证 | 20% | 是否满足性能需求 |
| 安全性 | 15% | 是否满足安全需求 |
| 可维护性 | 10% | 是否易于维护 |
| 文档完整性 | 10% | 文档是否完整 |

**评分标准**:
| 评分范围 | 等级 | 说明 |
|----------|------|------|
| 95-100 | S级 | 卓越，可直接使用 |
| 85-94 | A级 | 优秀，可使用 |
| 75-84 | B级 | 良好，需小改进 |
| 60-74 | C级 | 合格，需改进 |
| <60 | D级 | 不合格，需重大改进 |

| 超时类型 | 检测时机 | 处理策略 |

| 总迭代超时 | 每次迭代结束时检查总时间 | 终止迭代流程，保留当前产物，生成超时报告 |

## 四、错误处理

### 4.1 错误分级

| 错误级别 | 说明 | 处理策略 |
|----------|------|----------|
| 🔴 致命 | 流程无法继续 | 终止流程，返回错误报告 |
| 🟠 严重 | 步骤执行失败 | 重试或跳过，记录问题 |
| 🟡 警告 | 质量不达标 | 继续执行，标记改进点 |
| 🔵 信息 | 提示性信息 | 记录日志 |

### 4.2 常见错误

| 错误类型 | 级别 | 原因 | 处理方法 |
|----------|------|------|----------|
| 需求文档不存在 | 🔴 致命 | 文件路径错误 | 提示用户检查路径 |
| 需求分析失败 | 🟠 严重 | 需求文档格式不支持 | 返回原始文档，建议转换格式 |
| 设计生成失败 | 🟠 严重 | 需求规格不完整 | 返回需求分析步骤补充 |
| 验证不通过 | 🟡 警告 | 设计文档有缺陷 | 继续评审，记录问题 |
| 评审不达标 | 🟡 警告 | 评分<95 | 进入迭代改进流程 |
| 迭代超限 | 🔴 致命 | 迭代次数>6 | 终止流程，请求人工介入 |

### 4.3 回退机制

| 场景 | 回退策略 |
|------|----------|
| Step 1失败 | 终止流程，返回需求文档问题报告 |
| Step 2失败 | 回退到Step 1，补充需求信息 |
| Step 3失败 | 回退到Step 2，修改设计 |
| Step 4评分<95 | 进入Step 5迭代改进 |
| Step 5超限 | 终止流程，保留当前产物，请求人工介入 |
