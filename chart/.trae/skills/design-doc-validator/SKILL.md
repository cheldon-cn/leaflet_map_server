# Design Document Validator

设计文档质量检查与预处理Skill，确保设计文档满足task-planner的要求，产出高质量的任务拆分结果。

---

## 一、Skill概述

### 1.1 功能定位

本Skill用于在设计文档使用task-planner之前，进行质量检查和必要的预处理，确保：

1. **文档质量达标**：满足task-planner的输入要求
2. **问题提前发现**：识别影响任务拆分的问题
3. **预处理补充**：对不完整的文档进行结构化补充
4. **产出优化**：生成面向task-planner的结构化设计文档

### 1.2 Quick Start (5分钟)

**基本用法:**
```
> 检查xxx设计文档的质量
```

**验证后拆分任务:**
```
> 验证database_design.md后拆分任务
```

**That's it!** 系统会：
1. 解析设计文档结构
2. 执行5项必须项检查
3. 执行4项建议项检查
4. 生成检查清单报告
5. 如需预处理，自动补充缺失内容

**快速输出示例:**
```markdown
## 文档质量检查报告

### 质量等级: 🟢 A级（优秀）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| M1: 模块划分清晰 | ✅ 通过 | 识别出5个模块 |
| M2: 接口定义完整 | ✅ 通过 | 12个接口已定义 |
| M3: 依赖关系明确 | ✅ 通过 | 依赖图已绘制 |
| M4: 技术栈确定 | ✅ 通过 | Python 3.11 + FastAPI |
| M5: 文档一致性 | ✅ 通过 | 无冲突 |

### 建议: 可直接使用task-planner
```

### 1.3 性能预期

| 文档规模 | 字数 | 检查时间 | 输出大小 |
|----------|------|----------|----------|
| 小型 (<5000字) | 2000-5000 | 10-30秒 | ~2KB |
| 中型 (5000-15000字) | 5000-15000 | 30-60秒 | ~5KB |
| 大型 (15000-30000字) | 15000-30000 | 1-3分钟 | ~10KB |
| 超大型 (>30000字) | 30000+ | 3-5分钟 | ~15KB |

**性能影响因素:**
- 文档复杂度: 模块数量多会增加检查时间
- 问题数量: 问题多会增加预处理时间
- 预处理范围: 需要补充的内容多会增加时间

**优化建议:**
- 先进行快速预检查，识别主要问题
- 对于大型文档，可分段检查
- 预处理时优先处理P0问题

### 1.4 触发条件

当出现以下情况时，应调用本Skill：

- 用户需要对设计文档进行质量检查
- 用户准备使用task-planner但不确定文档是否合格
- 用户询问设计文档是否需要预处理
- 用户提到"设计文档检查"、"文档质量"、"预处理"等关键词

### 1.3 核心约束（必读）

**❌ 不要做的事**：
- 不要修改原始设计文档的意图
- 不要添加设计文档中未提及的功能
- 不要跳过任何必须项检查
- 不要在必须项未通过时跳过预处理
- 不要为不存在的模块编造接口
- 不要忽略用户提供的上下文信息

## Step 0: 输入验证

### 0.1 验证项

| 验证内容 | 通过条件 | 失败处理 |
|----------|----------|----------|
| 文档非空 | 文档长度 > 0 | 返回错误：文档为空，请提供有效设计文档 |
| 文档格式 | 可解析为文本 | 返回错误：文档格式不支持，请提供Markdown或文本格式 |
| 文档长度 | < 100,000 字符 | 提示：文档过大，建议分块检查 |

### 0.2 验证失败处理

当输入验证失败时：

1. **返回明确的错误信息**：说明验证失败的原因
2. **提供修复建议**：指导用户如何修复问题
3. **不继续执行后续步骤**：避免无效操作

```markdown
## 输入验证失败

**错误类型**: [空文档/格式不支持/文档过大]
**错误描述**: [详细描述]
**修复建议**: [具体建议]

请修复后重新调用本Skill。
```

## Step 2: 质量检查

### 2.1 必须满足项检查（🔴 阻塞项）

#### 检查项权重说明

| 检查项 | 权重 | 说明 |
|--------|------|------|
| M1 模块划分清晰 | 25% | 决定任务分解粒度，最关键 |
| M2 接口定义完整 | 25% | 决定任务依赖关系，最关键 |
| M3 依赖关系明确 | 20% | 决定任务顺序，重要 |
| M4 技术栈确定 | 15% | 决定技术复杂度，重要 |
| M5 文档一致性 | 15% | 确保文档质量，重要 |

#### 检查项 M1: 模块划分清晰

**检查标准**：
| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M1.1 | 是否识别出所有主要模块？ | 文档中明确列出≥1个模块 |
| M1.2 | 每个模块是否有明确的功能边界？ | 每个模块有功能描述 |
| M1.3 | 模块之间是否有清晰的职责划分？ | 无功能重叠或重叠已说明 |
| M1.4 | 是否明确了每个模块"不包含"的功能？ | 有明确边界说明（可选） |

**评分规则**：
- M1.1 + M1.2 + M1.3 全部通过 → ✅ 通过
- 任一项不通过 → ❌ 失败

**失败影响**：
- task-planner无法准确拆分任务
- 任务粒度过粗或过细
- 任务边界模糊

---

#### 检查项 M2: 接口定义完整

**检查标准**：
| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M2.1 | 是否列出了所有对外接口？ | 有接口列表或API说明 |
| M2.2 | 每个接口是否有输入参数定义？ | 有参数类型和说明 |
| M2.3 | 每个接口是否有输出/返回值定义？ | 有返回值类型和说明 |
| M2.4 | 是否定义了异常/错误处理？ | 有错误码或异常说明 |
| M2.5 | 是否有性能要求说明？ | 有性能指标（可选） |

**评分规则**：
- M2.1 + M2.2 + M2.3 + M2.4 全部通过 → ✅ 通过
- 任一项不通过 → ❌ 失败

**失败影响**：
- 无法拆分接口实现任务
- 无法定义验收标准
- 无法估算工时

---

#### 检查项 M3: 依赖关系明确

**检查标准**：
| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M3.1 | 是否识别了模块间的依赖关系？ | 有依赖关系说明 |
| M3.2 | 依赖关系是否有图表或文字说明？ | 有图表或清晰文字描述 |
| M3.3 | 是否识别了外部依赖（第三方库）？ | 有第三方库列表 |
| M3.4 | 是否分析了关键路径？ | 有关键路径说明（可选） |
| M3.5 | 是否存在循环依赖风险？ | 无循环依赖或已说明 |

**评分规则**：
- M3.1 + M3.2 + M3.3 全部通过 → ✅ 通过
- 任一项不通过 → ❌ 失败

**失败影响**：
- 无法确定任务顺序
- 无法识别并行任务
- 可能产生调度冲突

**评分规则**：
- M4.1 + M4.2 全部通过 → ✅ 通过

#### 检查项 M5: 文档一致性

**检查标准**：
| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M5.1 | 模块功能与接口一致性 | 模块描述的功能有对应接口 |

### 2.4 质量评分计算

```
质量评分公式：

必须项得分 = (通过项数 / 总项数) × 100
建议项得分 = (通过项数 / 总项数) × 100
可选项得分 = (通过项数 / 总项数) × 100

综合得分 = 必须项得分 × 0.6 + 建议项得分 × 0.3 + 可选项得分 × 0.1
```

**质量等级判定**：

| 等级 | 条件 | 说明 |
|------|------|------|
| 🟢 A级（优秀） | 必须项100% + 建议项≥75% | 可直接使用task-planner |
| 🟡 B级（合格） | 必须项100% + 建议项≥50% | 建议补充后使用 |
| 🟠 C级（待改进） | 必须项100% + 建议项<50% | 建议预处理后使用 |
| 🔴 D级（不合格） | 必须项<100% | 必须预处理后使用 |

## Step 3: 生成检查清单报告

### 3.1 报告生成规则

基于 `checklist_template.md` 模板，生成检查报告：

1. **更新检查统计表**：填写各分类的通过/失败/跳过项数
2. **更新质量等级**：根据评分结果判定
3. **更新问题汇总表**：汇总所有发现的问题
4. **更新各检查项状态**：标记每项的检查结果
5. **记录问题详情**：对每个问题详细描述

### 3.2 问题记录格式

```markdown
#### P01: [问题标题]

**问题分类**: [模块边界/接口定义/依赖关系/技术选型/数据模型/非功能需求/其他]

**问题描述**:
```

### 5.0 信息来源分类

预处理操作会补充文档中缺失的内容，这些信息的来源分为以下类型：

| 来源类型 | 说明 | 可靠性 | 标记 | 示例 |
|----------|------|--------|------|------|
| **原文提取** | 从原文档中提取 | 高 | 无需标记 | 模块名称、功能描述 |
| **行业推断** | 基于行业最佳实践推断 | 中 | ⚠️ | 接口参数类型、错误码、性能指标 |
| **询问用户** | 需要用户确认 | 待定 | ❓ | 技术选型、具体性能数值 |

```markdown
|----------|------|------|
```

## 三、负面约束（What NOT to Do）

### ❌ 范围约束

| 约束 | 原因 |
|------|------|

### ❌ 行为约束

| 约束 | 原因 |
|------|------|

| 不要生成过于复杂的依赖图 | 保持可读性 |
| 不要为不存在的模块编造接口 | 保持真实性 |
| 不要跳过问题记录步骤 | 确保可追溯性 |
