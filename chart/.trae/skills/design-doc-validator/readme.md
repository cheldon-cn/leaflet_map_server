## 快速开始

### 基本用法

```
> 检查[设计文档名].md的质量
> 验证[设计文档名].md是否符合要求
> 对[设计文档名].md进行预处理
```

### 完整示例

```
用户: 检查database_model_design.md的质量

AI: [调用design-doc-validator skill]
    [分析文档结构]
    [执行必须项检查]
    [执行建议项检查]
    [计算质量评分]
    
    检查结果:
    - 必须项: 5/5 通过 (100%)
    - 建议项: 3/4 通过 (75%)
    - 质量等级: 🟢 A级（优秀）
    
    可直接使用task-planner进行任务拆分。
```

## 功能概述

Design-Doc-Validator是一个设计文档质量检查与预处理Skill，确保：

| 功能 | 说明 |
|------|------|
| 质量检查 | 检查文档是否满足task-planner的输入要求 |
| 问题发现 | 识别影响任务拆分的问题 |
| 预处理补充 | 对不完整的文档进行结构化补充 |
| 产出优化 | 生成面向task-planner的结构化设计文档 |

### 触发关键词

| 关键词 | 示例 |

| 预处理 | "对设计文档进行预处理" |

| 验证文档 | "验证设计文档" |
| 文档检查 | "文档质量检查" |

### 适用场景

- ✅ 用户需要对设计文档进行质量检查
- ✅ 用户准备使用task-planner但不确定文档是否合格
- ✅ 用户询问设计文档是否需要预处理
- ✅ 用户提到"设计文档检查"、"文档质量"、"预处理"等关键词

### 不适用场景

- ❌ 用户需要拆分任务（建议使用task-planner）
- ❌ 用户需要技术评审（建议使用technical-reviewer）
- ❌ 用户需要多角色交叉评审（建议使用multi-role-reviewer）

## 检查项说明

### 必须满足项（🔴 阻塞项）

| 检查项 | 权重 | 说明 |
|--------|------|------|
| M1 模块划分清晰 | 25% | 决定任务分解粒度，最关键 |
| M2 接口定义完整 | 25% | 决定任务依赖关系，最关键 |
| M3 依赖关系明确 | 20% | 决定任务顺序，重要 |
| M4 技术栈确定 | 15% | 决定技术复杂度，重要 |
| M5 文档一致性 | 15% | 确保文档质量，重要 |

#### M1: 模块划分清晰

| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M1.1 | 是否识别出所有主要模块？ | 文档中明确列出≥1个模块 |
| M1.2 | 每个模块是否有明确的功能边界？ | 每个模块有功能描述 |
| M1.3 | 模块之间是否有清晰的职责划分？ | 无功能重叠或重叠已说明 |

#### M2: 接口定义完整

| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M2.1 | 是否列出了所有对外接口？ | 有接口列表或API说明 |
| M2.2 | 每个接口是否有输入参数定义？ | 有参数类型和说明 |
| M2.3 | 每个接口是否有输出/返回值定义？ | 有返回值类型和说明 |
| M2.4 | 是否定义了异常/错误处理？ | 有错误码或异常说明 |

#### M3: 依赖关系明确

| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M3.1 | 是否识别了模块间的依赖关系？ | 有依赖关系说明 |
| M3.2 | 依赖关系是否有图表或文字说明？ | 有图表或清晰文字描述 |
| M3.3 | 是否识别了外部依赖（第三方库）？ | 有第三方库列表 |

#### M4: 技术栈确定

| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M4.1 | 是否确定了编程语言？ | 有明确的语言说明 |
| M4.2 | 是否确定了核心框架？ | 有框架名称和版本 |

#### M5: 文档一致性

| 子项 | 检查内容 | 通过条件 |
|------|----------|----------|
| M5.1 | 模块功能与接口一致性 | 模块描述的功能有对应接口 |
| M5.2 | 依赖关系与模块定义一致性 | 依赖图中的模块在模块设计中存在 |
| M5.3 | 技术选型与实现要点一致性 | 选型的技术在实现要点中体现 |

## 质量评分

### 评分公式

```
必须项得分 = (通过项数 / 总项数) × 100
建议项得分 = (通过项数 / 总项数) × 100
可选项得分 = (通过项数 / 总项数) × 100

综合得分 = 必须项得分 × 0.6 + 建议项得分 × 0.3 + 可选项得分 × 0.1
```

### 质量等级

| 等级 | 条件 | 说明 |
|------|------|------|
| 🟢 A级（优秀） | 必须项100% + 建议项≥75% | 可直接使用task-planner |
| 🟡 B级（合格） | 必须项100% + 建议项≥50% | 建议补充后使用 |
| 🟠 C级（待改进） | 必须项100% + 建议项<50% | 建议预处理后使用 |
| 🔴 D级（不合格） | 必须项<100% | 必须预处理后使用 |

## 预处理决策

### 决策矩阵

| 必须项通过率 | 建议项通过率 | 预处理决策 | 工作量 |
|--------------|--------------|------------|--------|
| 100% | ≥75% | 不需要预处理 | - |
| 100% | 50%-74% | 可选预处理 | 小 |
| 100% | <50% | 建议预处理 | 中 |
| <100% | 任意 | 必须预处理 | 大 |

### 预处理操作

| 问题类型 | 预处理操作 | 优先级 |
|----------|------------|--------|
| 模块边界不清 | 补充模块功能范围 | P0 |
| 接口定义缺失 | 补充输入/输出/异常 | P0 |
| 依赖关系模糊 | 绘制依赖关系图 | P0 |
| 技术选型未定 | 确定技术栈 | P0 |
| 数据模型缺失 | 补充数据模型定义 | P1 |
| 非功能需求缺失 | 补充性能/安全要求 | P1 |

## 核心约束

**❌ 不要做的事**：

- 不要修改原始设计文档的意图
- 不要添加设计文档中未提及的功能
- 不要跳过任何必须项检查
- 不要在必须项未通过时跳过预处理
- 不要为不存在的模块编造接口
- 不要忽略用户提供的上下文信息

---

## 最佳实践

1. **在使用task-planner前先检查** - 确保文档质量达标
2. **关注必须项** - 必须项100%通过才能使用task-planner
3. **重视建议项** - 建议项影响任务拆分质量
4. **及时预处理** - 发现问题立即处理
5. **保存检查报告** - 便于后续跟踪和改进
