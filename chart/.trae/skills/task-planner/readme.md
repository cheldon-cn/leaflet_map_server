```
    [生成TASKS.md，包含里程碑、依赖关系和验收标准]
```

## 功能概述

Task Planner是一个将设计文档分解为独立、可执行开发任务的Skill，支持：

| 功能 | 说明 |
|------|------|
| 任务分解 | 将复杂模块拆分为4-8小时的小任务 |
| 依赖分析 | 自动识别任务间的依赖关系 |
| 关键路径 | 计算项目最短完成时间 |
| 资源分配 | 根据技能分配任务给开发者 |
| 工时估算 | 使用PERT方法进行三点估算 |
| 里程碑规划 | 按阶段组织任务 |

### 适用场景

- ✅ 用户提供了设计文档，需要创建开发计划
- ✅ 用户需要将复杂模块分解为可管理的任务
- ✅ 用户需要估算开发工作量和优先级
- ✅ 用户需要资源分配和团队容量规划

### 不适用场景

- ❌ 用户只有需求文档，没有设计文档（建议先使用design-doc-validator）
- ❌ 用户需要评审设计文档质量（建议使用technical-reviewer）
- ❌ 用户需要检查文档是否符合task-planner要求（建议使用design-doc-validator）

### 单个任务结构

```markdown
## Task: T1 - 项目初始化

### 描述
- 清晰描述需要完成的工作
- 技术方法和实现细节
- 与其他模块的集成点

### 优先级
- P0: 关键/阻塞（必须首先完成）
- P1: 高（核心功能）
- P2: 中（重要功能）
- P3: 低（锦上添花）

### 依赖
- 必须先完成的任务ID列表
- 外部依赖（库、API等）

### 验收标准
- [ ] **功能**: [具体可测量要求]
- [ ] **性能**: [可量化指标，如<100ms响应]
- [ ] **质量**: [具体质量门禁]
- [ ] **覆盖率**: 行覆盖率≥80%，分支覆盖率≥70%
- [ ] **文档**: API文档已生成，README已更新

### 工时估算（PERT方法）
- 乐观(O): 2h - 最佳情况
- 最可能(M): 4h - 正常情况
- 悲观(P): 8h - 最坏情况
- 期望值: 4.3h = (O + 4M + P) / 6
- 置信度: 高(>80%) / 中(50-80%) / 低(<50%)

### 资源需求
- 所需技能: [技能列表]
- 建议负责人: [开发者名称或角色]
- 可并行任务: [可同时进行的任务]

### 里程碑
- M1: 基础框架

### 状态
- 📋 待办 / 🔄 进行中 / 🔍 审核中 / ✅ 完成 / 🚫 阻塞
```

## 任务粒度指南

### 理想大小

| 指标 | 建议值 |
|------|--------|
| 理想工时 | 4-8小时（一个开发者，一天以内） |
| 最大工时 | 16小时（超过必须拆分） |
| 最小工时 | 2小时（更小则合并，除非关键） |

### 拆分时机

当任务满足以下条件时，应该拆分：
- 涉及多种技术/领域
- 有多个验收标准
- 跨多个组件
- 估算超过16小时
- 依赖关系不清晰

### 拆分方法

| 拆分维度 | 示例 |
|----------|------|
| 按组件 | 前端/后端/数据库 |
| 按层次 | API/服务/存储层 |
| 按功能 | 核心功能/边缘情况/错误处理 |
| 按测试 | 单元测试/集成测试/端到端测试 |

## 最佳实践

1. **单一职责**: 每个任务应该有一个明确目标
2. **可测试**: 每个任务必须有可验证的验收标准
3. **可估算**: 使用PERT方法提高准确性
4. **独立性**: 尽量减少任务间依赖
5. **有价值**: 每个任务应交付有形价值
6. **小粒度**: 任务理想大小4-8小时

## 常见问题

### Q1: 设计文档质量不高怎么办？

建议先使用`design-doc-validator`检查文档质量，进行必要的预处理后再使用task-planner。

### Q2: 任务估算不准确怎么办？

- 使用PERT三点估算法
- 记录实际工时与估算的差异
- 使用历史数据改进未来估算
- 定期回顾和调整估算

### Q3: 如何处理任务依赖冲突？

- 识别关键路径上的任务
- 优先完成关键路径任务
- 寻找可并行执行的任务
- 考虑资源重新分配

### Q4: 如何跟踪任务进度？

- 定期更新任务状态
- 使用状态工作流：待办→进行中→审核中→完成
- 跟踪团队速度（故事点/冲刺）
- 在里程碑节点进行回顾
