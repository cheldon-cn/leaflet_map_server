### 3.3 争议项讨论

#### 争议1: Sprint规划是否必须？

**争议描述**: 
- 敏捷专家认为Sprint规划是敏捷开发的核心，应该详细支持
- 软件工程任务规划师认为当前里程碑模式已足够，Sprint可作为扩展

**讨论过程**:
1. 敏捷专家解释：Scrum团队需要Sprint目标、Sprint Backlog、燃尽图
2. 规划师回应：里程碑模式适用于瀑布和混合模式，Sprint可作为可选扩展
3. AI专家建议：可以在Step 4后增加可选的Sprint规划章节

**最终决议**: 
- 权重从1.0降至0.75
- 作为可选扩展实现，不影响核心功能
- 状态：🟠 中优先级

#### 争议2: skill协作场景是否必要？

**争议描述**:
- 系统集成专家认为必须说明与其他skill的协作方式
- AI专家认为协作场景会增加复杂性，可能影响核心功能

**讨论过程**:
1. 集成专家解释：用户可能同时需要任务规划和评审
2. AI专家回应：过度协作可能导致职责不清
3. Prompt工程师建议：在"When to Invoke"中简要说明协作场景即可

**最终决议**:
- 权重从1.0降至0.88
- 在"When to Invoke"中增加协作场景说明
- 状态：🟠 中优先级

### 5.2 高优先级发现详情

#### P1: 缺少循环依赖检测机制

**状态**: 📋 待处理  
**提出者**: 软件工程任务规划师  
**权重**: 1.0

**问题描述**:
当任务A依赖任务B，任务B依赖任务C，任务C又依赖任务A时，形成循环依赖。当前skill没有提供检测和处理这种循环依赖的机制。

**影响范围**:
- 任务调度失败
- 项目进度阻塞
- 资源浪费

**修改方法**:
在Step 5（依赖图与关键路径分析）中增加循环依赖检测算法：
```markdown
#### Cycle Detection Algorithm

1. **Depth-First Search (DFS)**
   - Mark each node as: unvisited, visiting, visited
   - If we encounter a "visiting" node during DFS, cycle detected
   
2. **Cycle Resolution**
   - Identify all nodes in the cycle
   - Suggest breaking the cycle by:
     - Merging tasks
     - Removing unnecessary dependencies
     - Splitting tasks to break circular reference

3. **Visualization**
   - Highlight cycles in dependency graph with 🔴 CYCLE marker
   - Provide resolution suggestions
```

**验证方式**:
- 测试用例：创建包含循环依赖的任务图
- 预期：skill能够检测并报告循环依赖

#### R1: 缺少负面约束（NOT to do）

**状态**: 📋 待处理  
**提出者**: Prompt工程师  
**权重**: 0.88

**问题描述**:
当前skill只说明了应该做什么，没有明确说明不应该做什么。这可能导致AI产生超出范围的行为，如修改原始设计文档、添加未授权的功能等。

**影响范围**:
- AI可能修改原始设计文档
- AI可能添加设计文档中未提及的功能
- AI可能忽略用户的约束条件

**修改方法**:
增加"Negative Constraints"章节：
```markdown
## Negative Constraints (What NOT to Do)

To ensure focused and effective task planning, the skill must NOT:

### ❌ Scope Violations
| Constraint | Reason |
|------------|--------|
| Do NOT modify the original design document | Task planning should not change requirements |
| Do NOT add features not in design document | Stay within scope |
| Do NOT skip the user confirmation step | User must approve the plan |
| Do NOT estimate without understanding the task | Leads to inaccurate estimates |

### ❌ Behavior Violations
| Constraint | Reason |
|------------|--------|
| Do NOT create tasks larger than 16 hours | Violates granularity guidelines |
| Do NOT ignore dependencies between tasks | Causes scheduling issues |
| Do NOT proceed without critical path analysis | Misses project risks |
| Do NOT create orphan tasks | All tasks must have resolvable dependencies |
```

**验证方式**:
- 测试用例：检查AI是否遵守负面约束
- 预期：AI不会执行负面约束中列出的行为

**影响范围**:

```markdown
## Skill Selection Priority

When user request matches multiple skills, use this priority order:

| Priority | Skill | Trigger Keywords | When to Use |
|----------|-------|------------------|-------------|
| 1 | **User-specified** | User explicitly names a skill | Always honor user's explicit choice |
| 2 | **task-planner** | "任务规划", "任务分解", "task planning" | When user needs task breakdown |
| 3 | **technical-reviewer** | "评审", "审计", "review" | When user needs document review |

### Conflict Resolution
- If both task-planner and technical-reviewer match:
  - If user says "规划并评审": Run task-planner first, then technical-reviewer
  - If user says "评审并规划": Run technical-reviewer first, then task-planner
  - Default: Ask user which skill to invoke first
```

**验证方式**:
- 测试用例：用户请求同时匹配多个skill
- 预期：skill能够正确处理优先级冲突
