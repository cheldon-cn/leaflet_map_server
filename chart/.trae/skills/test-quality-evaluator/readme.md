## 概述

**Test Quality Evaluator** 是一个专为C++项目设计的测试质量评估和改进工具。它能够：

- ✅ 自动评估单元测试和集成测试质量
- ✅ 基于多维度评分体系生成详细报告
- ✅ 按优先级识别和修复测试问题
- ✅ 持续迭代改进直到达到质量标准（评分≥95）

### 为什么需要这个工具？

在软件开发过程中，测试质量往往难以量化评估。传统的测试覆盖率指标无法全面反映测试的真实质量。本工具提供：

1. **多维度评估** - 不仅关注覆盖率，还评估测试设计、独立性、可读性等
2. **标准化标准** - 基于`doc/tests_quality_evaluate.md`中的行业标准
3. **自动化改进** - 自动识别问题并实施改进措施
4. **持续迭代** - 循环评估和改进，直到达到目标质量

## 快速开始

### 前置条件

- 项目使用C++语言
- 测试文件位于`code/[modulename]/tests/`目录
- 评估标准文档位于`doc/tests_quality_evaluate.md`

### 基本用法

```
用户: 评估 geom 模块的测试质量
```

系统将自动：
1. 读取评估标准文档
2. 扫描测试文件
3. 评估所有维度
4. 生成评估报告
5. 如评分<95，自动改进并迭代

### 输出位置

评估报告将保存到：
```
code/[modulename]/tests/[modulename]_test_quality.md
```

## 评估维度

### 代码覆盖率标准

| 指标 | 优秀 (≥90%) | 良好 (80-89%) | 合格 (70-79%) | 不合格 (<70%) |
|------|-------------|---------------|---------------|---------------|
| 行覆盖率 | 核心模块 | 一般模块 | 需改进 | 严重缺失 |
| 分支覆盖率 | 所有分支 | 大部分分支 | 部分覆盖 | 缺少测试 |
| 函数覆盖率 | 100%公共函数 | ≥95% | ≥80% | 不足 |
| 类覆盖率 | 100% | ≥95% | ≥80% | 不完整 |

### 测试设计标准

#### 边界值测试

- [ ] 最小值、最小值-1
- [ ] 最大值、最大值+1
- [ ] 空值、空集合、空字符串
- [ ] 类型边界

#### 等价类划分

- [ ] 有效等价类
- [ ] 无效等价类
- [ ] 边界情况

#### 异常路径测试

- [ ] 无效输入参数
- [ ] 资源耗尽场景
- [ ] 错误状态恢复
- [ ] 错误码验证

#### 正负向测试比例

- 正向测试: 60-70%
- 负向测试: 30-40%

### 测试独立性标准

#### 隔离要求

- [ ] 无执行顺序依赖
- [ ] 无共享可变状态
- [ ] 测试可独立运行

#### 资源管理

- [ ] 正确使用SetUp/TearDown
- [ ] 无内存泄漏
- [ ] 环境状态恢复

### 测试可读性标准

#### 命名规范

- 模式: `MethodName_Scenario_ExpectedResult`
- 清晰表达测试意图
- 风格一致

#### 断言清晰性

- 使用语义明确的断言宏
- 断言消息描述期望值
- 一个测试验证一个关注点

#### 结构

- AAA模式 (Arrange-Act-Assert)
- 简洁明了
- 无复杂逻辑

### 测试维护性标准

#### 代码复用

- [ ] 公共逻辑抽取到辅助函数
- [ ] 合理使用测试夹具(Fixture)
- [ ] 避免代码重复

#### Mock使用

- [ ] 外部依赖正确Mock
- [ ] Mock行为符合预期
- [ ] 避免过度Mock

#### 数据管理

- [ ] 测试数据与代码分离
- [ ] 易于理解和修改
- [ ] 大型数据使用外部文件

## 最佳实践

### 1. 评估前准备

- ✅ 确保评估标准文档是最新的
- ✅ 测试文件命名符合规范
- ✅ 测试代码可编译运行

### 2. 改进过程

- ✅ 按优先级处理问题（P0 > P1 > P2 > P3）

- ✅ 综合评分 ≥ 95
- ✅ 所有P0和P1问题已解决

- ⚠️ 不要删除现有测试（除非有充分理由）

- ⚠️ 遵循项目现有的测试风格
- ⚠️ 评分达到95后停止，避免过度工程

## 常见问题

### Q1: 评分一直无法达到95怎么办？

**A**: 系统最多迭代6次。如果仍未达标，会建议人工介入。您可以：
1. 查看评估报告中的遗留问题
2. 手动解决关键问题
3. 重新运行评估

### Q2: 评估标准可以自定义吗？

**A**: 可以。修改`doc/tests_quality_evaluate.md`文件即可自定义评估标准、维度权重和阈值。

### Q3: 支持哪些测试框架？

**A**: 目前主要针对C++项目，支持常见的测试框架如Google Test、Catch2等。测试文件命名模式：
- 单元测试: `test_*.cpp` 或 `*_test.cpp`
- 集成测试: `test_it_*.cpp` 或 `*_integration_test.cpp`

### Q4: 如何与其他Skill配合使用？

**A**: 推荐的工作流程：
```
1. code-implementer → 实现代码
2. test-quality-evaluator → 评估测试
3. problem-tracker → 追踪问题
4. test-quality-evaluator → 重新评估
```

### Q5: 评估报告可以导出吗？

**A**: 评估报告以Markdown格式保存，可以：
- 直接在IDE中查看
- 转换为HTML或PDF
- 集成到CI/CD流程中
