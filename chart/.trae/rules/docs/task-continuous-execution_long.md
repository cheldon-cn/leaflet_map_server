## ⚠️ 关键约束摘要（AI必读）

> **重要**：以下约束是本规则的核心要求，AI执行时必须严格遵守。违反任何约束将导致规则执行失败。

### 核心约束（必须遵守）

| 约束ID | 约束内容 | 违反后果 |
|--------|----------|----------|
| **C1** | 必须持续执行直到所有任务完成或遇到阻塞 | ❌ 执行失败 |
| **C2** | 必须按优先级执行：P0→P1→P2→P3 | ❌ 执行顺序错误 |
| **C3** | 每个任务完成后必须自动检测继续 | ❌ 停止执行 |
| **C4** | 必须从配置文件读取所有参数 | ❌ 配置违规 |
| **C5** | 禁止执行一个任务后就停止 | ❌ 违反持续执行原则 |
| **C6** | 并发执行前必须检查依赖关系 | ❌ 依赖冲突 |
| **C7** | 执行前必须创建进度文件目录 | ❌ 文件写入失败 |
| **C8** | 每个任务完成后必须执行自检清单 | ❌ 质量失控 |

### 执行行为自检清单

**AI在每个任务完成后必须执行以下自检**：

```markdown
□ 任务状态已更新为 ✅ Done 或 ❌ Failed
□ 任务状态已写入 tasks.md 文件
□ 里程碑进度已更新
□ 性能数据已收集
□ 已扫描任务清单识别下一个任务
□ 如有待执行任务，已自动继续执行
□ 如无待执行任务，已输出完成报告
□ 如遇到阻塞，已输出阻塞报告并等待用户确认
□ 如用户请求停止，已保存进度并输出中断报告
```

### 配置优先级

```
环境变量 > .trae/config.json > 规则默认值
```

### 触发词匹配模式

- **精确匹配**：触发词必须完全匹配（默认）
- **模糊匹配**：触发词包含关键词即可
- **正则匹配**：使用正则表达式匹配

## 规则概述

本规则定义了Trae在执行任务清单时的持续执行机制，确保任务按优先级依次执行，直到所有任务完成，最大程度减少人工介入。

## 零、配置文件规范

### 0.1 配置文件位置

所有配置参数必须定义在 `.trae/config.json` 文件中。

### 0.2 配置文件强制执行

**强制要求**：规则执行时必须严格按照配置文件中的参数设置执行，不得使用硬编码默认值覆盖配置文件中的设置。

**配置加载流程**：

```markdown
Step 1: 启动时加载配置
  - 读取 .trae/config.json 文件
  - 解析 taskExecution 配置节
  - 验证配置项格式和有效性

Step 2: 配置验证
  IF 配置文件不存在 THEN
    → 使用默认配置
    → 输出警告: "配置文件不存在，使用默认配置"
  ELSE IF 配置项格式错误 THEN
    → 输出详细错误信息（见下方错误报告格式）
    → 使用默认配置
  ELSE
    → 使用配置文件中的设置
  END IF

Step 3: 配置应用
  - 所有执行参数必须从配置文件读取
  - 禁止在代码中硬编码覆盖配置值
  - 配置变更后自动重新加载
```

**配置验证错误报告格式**：

```markdown
## ⚠️ 配置验证失败

**错误类型**: [类型错误/范围错误/缺失字段/格式错误]
**配置文件**: .trae/config.json

### 错误详情

| 字段路径 | 错误类型 | 当前值 | 期望值 | 修复建议 |
|----------|----------|--------|--------|----------|
| limits.maxExecutionTime | 格式错误 | "abc" | 时间格式如"8h" | 修改为有效时间格式，如"8h"、"480m" |
| concurrency.maxConcurrent | 范围错误 | "10" | 1-5 | 并发数过大，建议设置为1-5之间 |
| validation.validStatuses | 缺失字段 | - | 状态列表 | 添加有效状态列表 |

### 修复示例

**错误字段**: limits.maxExecutionTime
**当前值**: "abc"
**修复后**:
```json
{
  "limits": {
    "maxExecutionTime": "8h"
  }
}
```

### 使用默认配置

由于配置验证失败，将使用以下默认值：
- maxExecutionTime: "8h"
- maxConcurrent: 3
- ...

请修正配置文件后重新执行。
```

### 0.3 配置项说明

| 配置节 | 说明 | 必需 |
|--------|------|------|
| triggerKeywords | 触发关键词列表 | ✅ |
| limits | 执行限制参数 | ✅ |
| concurrency | 并发执行配置 | ✅ |
| performance | 性能监控配置 | ✅ |
| stopKeywords | 停止请求关键词 | ✅ |
| paths | 文件路径配置 | ✅ |
| validation | 格式验证配置 | ✅ |
| retry | 重试机制配置 | ✅ |
| reporting | 报告输出配置 | ✅ |

### 0.4 配置优先级

```markdown
配置优先级（从高到低）：
1. 环境变量（TASK_EXEC_前缀）
2. .trae/config.json 配置文件
3. 规则文档中的默认值

禁止行为：
- ❌ 跳过配置文件直接使用默认值
- ❌ 硬编码覆盖配置文件设置
- ❌ 忽略配置文件中的任何配置项
```

### 0.5 环境变量覆盖机制

**环境变量命名规则**：

| 配置路径 | 环境变量名 | 示例值 |
|----------|------------|--------|
| limits.maxExecutionTime | TASK_EXEC_MAX_TIME | "8h" |
| limits.maxTasks | TASK_EXEC_MAX_TASKS | "100" |
| limits.maxMemory | TASK_EXEC_MAX_MEMORY | "2GB" |
| concurrency.enabled | TASK_EXEC_CONCURRENCY | "true" |
| concurrency.maxConcurrent | TASK_EXEC_MAX_CONCURRENT | "3" |
| performance.enableMonitoring | TASK_EXEC_MONITORING | "true" |

**环境变量加载流程**：

```markdown
Step 1: 扫描环境变量
  FOR 每个配置项 DO
    IF 存在对应的环境变量 THEN
      → 读取环境变量值
      → 转换为配置项类型
      → 覆盖配置文件中的值
    END IF
  END FOR

Step 2: 验证环境变量
  IF 环境变量值格式错误 THEN
    → 输出警告: "环境变量 {VAR_NAME} 格式错误，使用配置文件值"
    → 忽略该环境变量
  END IF

Step 3: 应用配置
  → 使用优先级最高的配置值
```

**示例**：

```bash
# 设置环境变量覆盖配置
export TASK_EXEC_MAX_TIME="4h"
export TASK_EXEC_MAX_CONCURRENT="5"
export TASK_EXEC_CONCURRENCY="false"

# 执行任务清单
# 将使用：最大执行时间4小时，禁用并发执行
```

## 一、触发条件

### 1.1 默认触发词

当用户发送以下提示词时，本规则自动生效：

- "按任务清单tasks.md执行"
- "按照tasks.md清单实施"
- "继续按任务清单执行"
- "实施任务清单"
- "执行任务清单"

### 1.2 自定义触发词

支持在项目配置文件 `.trae/config.json` 中添加自定义触发词：

```json
{
  "taskExecution": {
    "triggerKeywords": [
      "开始任务",
      "执行计划",
      "run tasks",
      "continue tasks"
    ]
  }
}
```

### 1.3 触发词匹配规则

**匹配模式配置**：

在 `.trae/config.json` 中配置匹配模式：

```json
{
  "taskExecution": {
    "triggerMatchMode": "exact",  // "exact" | "fuzzy" | "regex"
    "triggerKeywords": [...]
  }
}
```

| 匹配模式 | 说明 | 示例 |
|----------|------|------|
| **exact** | 精确匹配（默认） | "按任务清单tasks.md执行" 必须完全匹配 |
| **fuzzy** | 模糊匹配 | 包含 "任务清单" 或 "执行" 即可触发 |
| **regex** | 正则表达式匹配 | 使用正则表达式定义触发模式 |

**正则匹配示例**：

```json
{
  "taskExecution": {
    "triggerMatchMode": "regex",
    "triggerPatterns": [
      "^(按|按照|继续).*tasks?\\.md.*执行$",
      "^(执行|实施|开始).*任务.*清单$",
      "^run\\s+tasks?$"
    ]
  }
}
```

**匹配优先级**：

```markdown
IF triggerMatchMode == "exact" THEN
  → 用户输入必须完全匹配触发词
ELSE IF triggerMatchMode == "fuzzy" THEN
  → 用户输入包含任一触发词关键词即可
ELSE IF triggerMatchMode == "regex" THEN
  → 用户输入匹配任一正则表达式即可
END IF
```

## 二、核心执行原则

### 2.1 持续执行原则

**强制要求**：Trae必须持续执行任务清单，直到满足以下任一条件：

1. 所有任务状态为 `✅ Done` 或 `已完成`
2. 遇到阻塞问题需要用户确认
3. 用户明确要求停止

**禁止行为**：

- ❌ 执行一个或多个任务后自动终止
- ❌ 未检测任务完成状态就停止执行
- ❌ 需要用户多次发送"继续执行"提示词

### 2.2 优先级执行规则

任务执行顺序必须遵循以下优先级：

```
P0 (关键/阻塞) → P1 (高) → P2 (中) → P3 (低)
```

**同级任务执行顺序**：

1. 按依赖关系：先执行被依赖的任务
2. 按关键路径：优先执行关键路径上的任务
3. 按里程碑：按M1 → M2 → M3 → M4 → M5顺序

### 2.3 任务状态检测机制

**每次任务执行完成后，必须执行以下检测**：

```markdown
## 任务完成检测流程

Step 1: 更新当前任务状态
  - 将当前任务状态更新为 ✅ Done 或 ❌ Failed
  - 记录完成时间和实际工时
  - 记录执行结果和输出

Step 2: 扫描任务清单
  - 统计所有任务状态
  - 识别下一个待执行任务

Step 3: 判断是否继续
  IF 存在状态为 📋 Todo 或 ⚠️ Partial 的任务 THEN
    → 自动继续执行下一个任务
    → 无需用户介入
  ELSE IF 存在状态为 ❌ Failed 的任务 THEN
    → 输出失败报告
    → 等待用户确认是否重试或跳过
  ELSE
    → 所有任务已完成
    → 输出完成报告
    → 终止执行
  END IF
```

### 2.4 执行限制机制

**强制限制**：

| 限制项 | 默认值 | 说明 | 可配置 |
|--------|--------|------|--------|
| 最大执行时间 | 8小时 | 防止长时间运行 | ✅ |
| 最大任务数 | 100个 | 防止任务过多 | ✅ |
| 最大循环次数 | 1000次 | 防止无限循环 | ✅ |
| 内存使用上限 | 2GB | 防止内存溢出 | ✅ |
| 连续失败上限 | 5个 | 防止持续失败 | ✅ |
| **单任务超时** | **10分钟** | **单个任务最大执行时间** | ✅ |

**配置方式**：

在 `.trae/config.json` 中配置：

```json
{
  "taskExecution": {
    "limits": {
      "maxExecutionTime": "8h",
      "maxTasks": 100,
      "maxLoops": 1000,
      "maxMemory": "2GB",
      "maxConsecutiveFailures": 5,
      "maxWaitTimeForCurrentTask": "5m",
      "taskTimeout": "10m"
    }
  }
}
```

**执行前初始化**：

```markdown
Step 1: 创建必要目录
  - 创建 .trae/progress/ 目录（如果不存在）
  - 创建 .trae/logs/ 目录（如果不存在）
  - 验证目录写入权限

Step 2: 初始化进度文件
  - 创建空的进度文件（如果不存在）
  - 写入初始状态

Step 3: 验证配置
  - 验证所有配置项格式
  - 验证目录权限
  - 验证任务清单文件存在
```

**超时处理流程**：

```markdown
Step 1: 检测是否超限
  IF 执行时间超过最大限制 OR 循环次数超过限制 OR 内存超过上限 THEN
    → 触发超时处理
  END IF

Step 2: 保存当前进度
  - 保存已完成任务列表
  - 保存当前任务状态
  - 保存执行统计信息

Step 3: 输出超时报告
  - 说明超时原因
  - 显示当前进度
  - 提供继续执行的命令

Step 4: 等待用户确认
  - 用户可选择继续执行或终止
  - 继续执行时重置计数器
```

**连续失败处理**：

```markdown
IF 连续失败任务数 >= 最大连续失败上限 THEN
  → 输出失败报告
  → 分析失败原因
  → 等待用户确认是否继续
END IF
```

### 2.5 任务清单格式验证

**必需字段**：

| 字段 | 格式 | 示例 | 说明 |
|------|------|------|------|
| 任务ID | 字母+数字 | T1, T2, T10 | 唯一标识 |
| 任务名称 | 文本 | 几何模块核心类实现 | 简洁描述 |
| 任务状态 | Emoji+文本 | 📋 Todo, ✅ Done | 当前状态 |
| 优先级 | P+数字 | P0, P1, P2, P3 | 执行优先级 |

**可选字段**：

| 字段 | 格式 | 示例 | 说明 |
|------|------|------|------|
| 依赖关系 | 任务ID列表 | T1, T2 | 前置任务 |
| 预估工时 | 数字+单位 | 8h, 2d | 工作量估算 |
| 负责人 | 文本 | Dev A | 任务负责人 |
| 里程碑 | M+数字 | M1, M2 | 所属里程碑 |
| 验收标准 | 列表 | 功能可用，测试通过 | 完成标准 |

**格式验证流程**：

```markdown
Step 1: 文件存在性检查
  IF 任务清单文件不存在 THEN
    → 输出错误: "任务清单文件不存在"
    → 等待用户提供文件路径
  END IF

Step 2: 格式类型检测
  IF 文件扩展名为 .json THEN
    → 使用JSON解析器
  ELSE IF 文件扩展名为 .yaml 或 .yml THEN
    → 使用YAML解析器
  ELSE
    → 使用Markdown解析器（默认）
  END IF

Step 3: 必需字段验证
  FOR 每个任务 DO
    IF 缺少必需字段 THEN
      → 记录错误: 任务ID, 缺少字段
    END IF
  END FOR

Step 4: 数据一致性验证
  - 任务ID唯一性检查
  - 依赖关系循环检查
  - 优先级有效性检查
  - 状态有效性检查

Step 5: 输出验证报告
  IF 存在错误 THEN
    → 输出详细错误列表
    → 提供修复建议
    → 等待用户修正
  ELSE
    → 验证通过，开始执行
  END IF
```

**格式错误报告示例**：

```markdown
## 任务清单格式验证失败

### 错误列表

| 行号 | 任务ID | 错误类型 | 错误描述 |
|------|--------|----------|----------|
| 15 | T5 | 缺少字段 | 缺少"优先级"字段 |
| 23 | T8 | 格式错误 | 优先级格式错误，应为P0-P3 |
| 31 | T10 | 依赖错误 | 依赖任务T99不存在 |
| 45 | T15 | 循环依赖 | T15依赖T20，T20依赖T15 |

### 修复建议

1. 任务T5: 添加优先级字段，例如: `| P1 |`
2. 任务T8: 修改优先级为P0、P1、P2或P3
3. 任务T10: 检查依赖任务ID是否正确
4. 任务T15和T20: 解除循环依赖关系

请修正后重新执行。
```

### 3.3 并发执行策略

**并发判断规则**：

```markdown
Step 1: 依赖关系检查
  FOR 每个候选并发任务 DO
    IF 任务有依赖关系 THEN
      → 检查依赖任务是否已完成
      → IF 依赖未完成 THEN
        → 从并发候选列表中移除该任务
      END IF
    END IF
  END FOR

Step 2: 并发可行性判断
  IF 候选任务数 >= 2 AND 无依赖冲突 AND 资源充足 THEN
    → 支持并发执行
    → 最大并发数: 配置值（默认3）
  ELSE
    → 串行执行
  END IF
```

**并发控制机制**：

| 机制 | 说明 | 实施方式 |
|------|------|----------|
| 任务锁 | 防止同一任务被多次执行 | 任务ID加锁，执行完成后释放 |
| 资源限制 | 控制并发任务的总资源消耗 | 监控内存、CPU，动态调整并发数 |
| 失败隔离 | 单个任务失败不影响其他并发任务 | 独立错误处理，记录失败任务 |
| 状态同步 | 并发任务状态更新互不干扰 | 文件锁或队列机制 |
| **依赖检查** | **并发前验证依赖关系** | **有依赖的任务强制串行执行** |
| **资源隔离** | **并发任务独立资源上下文** | **独立工作目录、环境变量、临时文件** |

**资源隔离机制**：

```markdown
为每个并发任务创建独立资源上下文：

1. 工作目录隔离
   - 主工作目录: .trae/workspace/
   - 任务工作目录: .trae/workspace/task_{id}_{timestamp}/
   - 任务完成后清理工作目录

2. 环境变量隔离
   - 每个任务继承主环境变量
   - 添加任务特定环境变量: TASK_ID, TASK_WORKDIR

3. 临时文件隔离
   - 任务临时目录: .trae/temp/task_{id}/
   - 任务完成后自动清理

4. 输出文件隔离
   - 任务输出目录: .trae/output/task_{id}/
   - 合并到主输出目录前进行冲突检查
```

**依赖冲突检测**：

```markdown
检测规则：
IF 任务A依赖任务B AND 任务B在并发列表中 THEN
  → 依赖冲突
  → 解决方案：任务A等待任务B完成后执行
END IF

示例：
T1 (P0) → 完成
T2 (P0, 依赖T1) → 等待T1完成
T3 (P0, 无依赖) ┐
T4 (P0, 无依赖) ├─ 并发执行（T2不在并发列表中）
T5 (P0, 无依赖) ┘
T6 (P1, 依赖T3) → 等待T3完成
```

**并发执行流程**：

```
┌─────────────────────────────────────────────────────────────┐
│              并发执行流程                                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 识别可并发任务                                       │
│  ├── 扫描任务清单                                             │
│  ├── 筛选优先级相同的任务                                      │
│  └── 检查依赖关系（无依赖）                                    │
│                                                             │
│  Step 2: 资源评估                                            │
│  ├── 检查当前内存使用                                         │
│  ├── 检查当前CPU使用率                                        │
│  └── 计算可用并发槽位                                         │
│                                                             │
│  Step 3: 启动并发任务                                         │
│  ├── 分配任务槽位                                             │
│  ├── 启动任务执行                                             │
│  └── 监控执行状态                                             │
│                                                             │
│  Step 4: 并发监控                                            │
│  ├── 监控资源使用                                             │
│  ├── 检测任务完成                                             │
│  └── 动态调整并发数                                           │
│                                                             │
│  Step 5: 任务完成处理                                         │
│  ├── 更新任务状态                                             │
│  ├── 释放任务槽位                                             │
│  └── 启动下一批任务                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**并发风险控制**：

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 资源竞争 | 多个任务争抢资源 | 资源监控，动态调整并发数 |
| 状态冲突 | 并发写入任务状态 | 文件锁或队列机制 |
| 依赖冲突 | 任务依赖关系错误 | 严格检查依赖关系 |
| 失败传播 | 一个任务失败影响其他 | 失败隔离，独立错误处理 |

## 七、异常处理

### 7.1 阻塞问题分类

**技术阻塞**：

| 阻塞类型 | 判断标准 | 处理方式 |
|----------|----------|----------|
| 依赖缺失 | 缺少必要的库、工具或环境 | 等待用户安装或配置 |
| 编译失败 | 代码编译或构建失败 | 等待用户修复代码 |
| 测试环境 | 测试环境不可用或配置错误 | 等待用户修复环境 |
| 网络问题 | 网络连接失败或超时 | 自动重试3次，失败后等待用户 |
| 权限不足 | 缺少必要的权限 | 等待用户授权 |

**业务阻塞**：

| 阻塞类型 | 判断标准 | 处理方式 |
|----------|----------|----------|
| 需求不明确 | 任务描述模糊或矛盾 | 等待用户澄清需求 |
| 设计缺失 | 缺少必要的设计文档 | 等待用户提供设计文档 |
| 验收冲突 | 验收标准之间存在冲突 | 等待用户确认优先级 |
| 依赖任务失败 | 依赖的任务执行失败 | 等待用户决定是否继续 |

**资源阻塞**：

| 阻塞类型 | 判断标准 | 处理方式 |
|----------|----------|----------|
| 内存不足 | 内存使用超过上限 | 等待用户释放资源或调整限制 |
| 磁盘不足 | 磁盘空间不足 | 等待用户清理磁盘 |
| CPU过载 | CPU使用率持续过高 | 等待用户优化或调整并发数 |

**阻塞判断标准**：

```markdown
IF 问题无法通过自动重试解决 AND 需要外部资源或决策 THEN
  → 识别为阻塞问题
  → 分类阻塞类型
  → 输出阻塞报告
  → 等待用户确认
END IF
```

### 7.3 用户停止请求处理

**停止请求识别**：

用户可通过以下方式请求停止执行：

| 停止请求关键词 | 说明 |
|----------------|------|
| "停止执行" | 立即停止任务执行 |
| "取消任务" | 取消当前任务 |
| "终止" | 终止整个执行流程 |
| "暂停" | 暂停执行，可恢复 |
| "中断" | 中断当前操作 |

**停止处理流程**：

```markdown
Step 1: 识别停止请求
  IF 用户发送停止请求 THEN
    → 记录停止时间
    → 识别停止类型（停止/暂停/中断）
  END IF

Step 2: 完成当前任务
  IF 当前有正在执行的任务 THEN
    → 等待当前任务完成（最多等待5分钟）
    → 或强制中断（用户选择"强制停止"）
  END IF

Step 3: 保存执行进度
  - 保存已完成任务列表
  - 保存当前任务状态
  - 保存执行统计信息
  - 保存性能数据
  - 写入进度文件: .trae/progress/task_execution_progress.json

Step 4: 输出中断报告
  - 说明中断原因
  - 显示当前进度
  - 提供恢复命令

Step 5: 终止执行
  - 释放所有资源
  - 关闭所有连接
  - 清理临时文件
  - 终止执行流程
```

本规则检测任务完成状态
    │
    ├─ 有待执行任务 → 继续调用 code-implementer
    │
    └─ 无待执行任务 → 输出完成报告，终止执行
```

|------|------|

| code-implementer Skill | 单个任务的TDD实施、测试验证、代码提交 |

## 十一、规则验证

### 11.1 详细验证清单

**执行前验证**：

- [ ] 任务清单文件存在且可读
- [ ] 任务清单格式符合要求
- [ ] 所有任务ID唯一
- [ ] 依赖关系无循环
- [ ] 优先级定义正确（P0-P3）
- [ ] 任务状态有效（📋/🔄/✅/⚠️/🚫/❌）
- [ ] 必需字段完整

**执行中验证**：

- [ ] 是否按优先级执行任务
- [ ] 是否在每个任务完成后自动检测继续
- [ ] 是否正确处理阻塞问题
- [ ] 是否自动更新任务状态和进度统计
- [ ] 是否遵守执行限制（时间、循环次数、内存）
- [ ] 是否收集性能数据
- [ ] 是否正确处理失败任务
- [ ] 是否在连续失败时停止

**执行后验证**：

- [ ] 是否在所有任务完成后输出最终报告
- [ ] 最终报告内容完整
- [ ] 所有任务状态正确
- [ ] 性能指标在合理范围
- [ ] 性能统计已包含在报告中

**性能验证**：

- [ ] 任务执行时间 < 5分钟（平均）
- [ ] 状态检测时间 < 100ms
- [ ] 内存使用 < 500MB
- [ ] CPU使用率 < 80%
- [ ] 任务成功率 > 95%

### 11.2 失败条件

以下情况视为规则执行失败：

- 执行一个或多个任务后自动终止，未检测完成状态
- 需要用户多次发送"继续执行"提示词
- 未按优先级执行任务
- 未自动更新任务状态

### 14.3 故障排查指南

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| 任务执行卡住 | 遇到阻塞问题 | 查看阻塞报告，按提示处理 |
| 内存持续增长 | 任务清单过大 | 启用缓存，减少任务数 |
| 执行时间过长 | 任务复杂度高 | 调整最大执行时间限制 |
| 连续任务失败 | 环境或依赖问题 | 检查错误日志，修复环境 |
| 状态更新失败 | 文件权限问题 | 检查文件权限，重新授权 |
