## 一、文档概述

### 1.1 目的

本文档详细定义海图显示系统的用户界面布局设计，包括：
- 整体布局架构
- 各组件详细规格
- 交互行为定义
- 技术实现方案
- 性能与扩展要求

### 1.2 适用范围

本设计适用于：
- 海图显示系统主界面
- 预警系统集成
- 航线规划功能
- AIS监控功能

### 1.3 目标用户

| 用户角色 | 特点 | 主要需求 |
|----------|------|----------|
| 航海人员 | 操作熟练度一般 | 直观界面、快速预警响应 |
| 海事管理员 | 操作熟练 | 高效工具、批量处理 |
| 系统管理员 | 技术背景 | 配置能力、故障排查 |

## 三、布局架构设计

### 3.1 整体布局架构

采用 **Ribbon + VSCode混合布局（方案B：右侧面板布局）**：

```
┌──────────────────────────────────────────────────────────────────────────┐
│                           Ribbon 菜单栏 (高度: 100-120px)                 │
│  ┌─────────────────────────────────────────────────────────────────────┐ │
│  │ [文件] [视图] [工具] [图层] [航线] [预警] [设置] [帮助]              [左侧栏][右侧栏][用户] [最小化][最大化][关闭]│ │
│  ├─────────────────────────────────────────────────────────────────────┤ │
│  │  功能按钮组...                                                       │ │
│  └─────────────────────────────────────────────────────────────────────┘ │
├────────┬──────────────────────┬─────────────────────────┬────────────────┤
│ 活动栏  │     侧边栏面板       │       海图显示区         │   右侧面板     │
│ (48px)  │   (可折叠,可拖拽)    │     (Canvas+交互层)     │ (可折叠,可拖拽)│
│        │    宽度: 200-300px   │                         │ 宽度: 200-400px│
│ [📁]   │ ┌─────────────────┐ │                         │ [预警] [日志]  │
│ [🔍]   │ │图层管理/数据目录│ │                         │ [属性] [终端]  │
│ [🛤️]   │ │/航线管理/预警配置│ │                         │                │
│ [⚠️]   │ └─────────────────┘ │                         │ 预警列表       │
│ [📊]   │ (与活动栏图标关联)   │                         │ 操作日志       │
│ [⚙️]   │                      │                         │ 要素属性       │
│        │                      │                         │ 命令行         │
├────────┴──────────────────────┴─────────────────────────┴────────────────┤
│                           状态栏 (高度: 22px)                              │
│  [服务状态] [坐标] [比例尺] [图层] [预警数] [时间]           [通知] [设置] │
└──────────────────────────────────────────────────────────────────────────┘
```

### 3.2 布局组件概览

| 组件 | 位置 | 尺寸规格 | 特性 |
|------|------|----------|------|
| Ribbon菜单栏 | 顶部 | 高度100-120px | 固定高度，标签页切换 |
| 活动栏 | 最左侧 | 宽度48px | 固定宽度，图标按钮 |
| 侧边栏面板 | 活动栏右侧 | 宽度200-300px | 可折叠、可拖拽调整 |
| 海图显示区 | 中央 | 自适应 | Canvas + 交互层 |
| 右侧面板 | 海图显示区右侧 | 宽度200-400px | 可折叠、可拖拽调整 |
| 状态栏 | 底部 | 高度22px | 固定高度 |

### 3.3 技术架构

```
MainView (BorderPane)
├── Top: Ribbon (fxribbon模块)
├── Center: SplitPane (水平分割)
│   ├── Left: ActivityBar + SideBar
│   │   ├── ActivityBar (VBox, 48px宽)
│   │   └── SideBar (StackPane, 可折叠, 200-300px)
│   ├── Center: ChartCanvas (海图显示)
│   └── Right: RightPanel (可折叠, 200-400px)
│       └── TabPane (预警/日志/属性/终端)
└── Bottom: StatusBar (HBox, 22px高)
```

### 3.4 布局选择理由

| 评估维度 | 方案B优势 |
|----------|-----------|
| 海图显示空间 | 高度最大化，符合海图浏览习惯 |
| 信息展示效率 | 纵向列表展示更符合阅读习惯 |
| 预警监控 | 右侧面板可同时展示预警列表和详情 |
| 操作便捷性 | 侧边信息面板与主显示区并列，视线无需上下切换 |
| 布局灵活性 | 左右面板均可独立拖拽调整宽度 |

### 3.5 方案综合评估

#### 3.5.1 评估维度定义

| 维度 | 权重 | 说明 |
|------|------|------|
| **海图显示空间** | 25% | 主显示区占比 |
| **功能可发现性** | 20% | 新用户能否快速找到功能 |
| **操作效率** | 20% | 熟练用户的操作速度 |
| **界面美观度** | 15% | 现代感、专业感 |
| **实现复杂度** | 10% | 开发工作量 |
| **可扩展性** | 10% | 未来功能扩展能力 |

#### 3.5.2 方案评分对比

| 方案 | 显示空间 | 可发现性 | 操作效率 | 美观度 | 复杂度 | 扩展性 | **加权总分** |
|------|----------|----------|----------|--------|--------|--------|--------------|
| 方案一(纯VSCode) | 95 | 60 | 85 | 90 | 70 | 85 | **80.25** |
| 方案二(Ribbon+VSCode) | 80 | 90 | 80 | 85 | 60 | 90 | **82.00** |
| 方案三(可折叠Ribbon) | 88 | 85 | 82 | 88 | 40 | 85 | **80.70** |
| 方案四(传统工具栏) | 75 | 85 | 75 | 70 | 80 | 75 | **76.25** |

**评分说明**:
- **显示空间**: 方案一得分最高(95)，因为无Ribbon占用空间；方案二因Ribbon占用空间得分较低(80)
- **可发现性**: 方案二得分最高(90)，Ribbon菜单功能分类清晰，新用户易上手
- **操作效率**: 方案一得分最高(85)，快捷键+命令面板适合熟练用户
- **美观度**: 方案一和方案三得分较高，符合现代UI趋势
- **复杂度**: 方案三得分最低(40)，可折叠Ribbon实现复杂
- **扩展性**: 方案二得分最高(90)，Ribbon+面板组合扩展性强

**推荐方案**: 方案二(Ribbon+VSCode混合布局)，加权总分82.00分，兼顾新用户可发现性和熟练用户操作效率。

#### 3.5.3 场景适配分析

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| **航海人员日常使用** | 方案（右侧面板） | 直观易用，预警信息纵向展示更清晰 |
| **海事管理专业操作** | 方案（右侧面板） | 显示空间大，操作效率高 |
| **预警系统监控** | 方案（右侧面板） | 右侧面板纵向展示预警信息更高效 |
| **多用户混合场景** | 方案（右侧面板） | 灵活适应不同显示器尺寸 |

## 七、海图显示区设计

### 7.1 布局规格

| 属性 | 值 |
|------|-----|
| 位置 | 中央区域 |
| 尺寸 | 自适应填充剩余空间 |

### 7.2 渲染架构

```
ChartCanvas (分层渲染)
├── 背景层 (BackgroundLayer)
│   └── 海洋、陆地底图
├── 海图层 (ChartLayer)
│   ├── 栅格海图
│   └── 矢量海图
├── 图层叠加 (OverlayLayers)
│   ├── 航线层
│   ├── AIS目标层
│   ├── 预警区域层
│   └── 标注层
├── 交互层 (InteractionLayer)
│   ├── 选中高亮
│   ├── 悬停效果
│   └── 绘制临时图形
└── 信息层 (InfoLayer)
    ├── 比例尺
    ├── 指北针
    └── 坐标网格
```

#### 7.2.1 渲染数据流

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          渲染数据流向                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  数据源                  渲染层                    显示                      │
│  ┌─────────┐            ┌─────────┐              ┌─────────┐               │
│  │ 海图数据 │ ─────────▶ │ 背景层  │ ──────────▶ │         │               │
│  └─────────┘            └─────────┘              │         │               │
│  ┌─────────┐            ┌─────────┐              │         │               │
│  │ 图层数据 │ ─────────▶ │ 海图层  │ ──────────▶ │         │               │
│  └─────────┘            └─────────┘              │         │               │
│  ┌─────────┐            ┌─────────┐              │ Canvas  │               │
│  │ 航线数据 │ ─────────▶ │ 叠加层  │ ──────────▶ │         │               │
│  └─────────┘            └─────────┘              │         │               │
│  ┌─────────┐            ┌─────────┐              │         │               │
│  │ 交互事件 │ ─────────▶ │ 交互层  │ ──────────▶ │         │               │
│  └─────────┘            └─────────┘              │         │               │
│  ┌─────────┐            ┌─────────┐              │         │               │
│  │ 视图状态 │ ─────────▶ │ 信息层  │ ──────────▶ │         │               │
│  └─────────┘            └─────────┘              └─────────┘               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 7.2.2 层间通信机制

| 层级 | 通信方式 | 说明 |
|------|----------|------|
| 背景层 → 海图层 | 数据引用 | 共享地理坐标范围 |
| 海图层 → 叠加层 | 事件通知 | 海图缩放时通知叠加层更新 |
| 叠加层 → 交互层 | 坐标转换 | 将屏幕坐标转换为地理坐标 |
| 交互层 → 信息层 | 状态同步 | 同步当前视图状态 |

#### 7.2.3 渲染优先级

| 优先级 | 层级 | 渲染顺序 | 说明 |
|--------|------|----------|------|
| 1 (最低) | 背景层 | 最先渲染 | 作为底图，不响应交互 |
| 2 | 海图层 | 第二渲染 | 海图要素，响应选择交互 |
| 3 | 叠加层 | 第三渲染 | 业务数据，响应全部交互 |
| 4 | 交互层 | 第四渲染 | 临时图形，高优先级显示 |
| 5 (最高) | 信息层 | 最后渲染 | 始终在最上层，不遮挡 |

## 十三、用户旅程设计

### 12.1 预警响应流程

```
阶段1: 预警发现
├── 触发: 状态栏预警图标闪烁 + 声音提示
├── 操作: 用户注意到预警
└── 期望: 预警信息明显但不干扰当前工作

阶段2: 预警查看
├── 触发: 点击状态栏预警图标 或 右侧面板预警标签
├── 操作: 查看预警列表和详情
└── 期望: 快速了解预警类型、位置、严重程度

阶段3: 预警定位
├── 触发: 点击预警项
├── 操作: 海图自动定位到预警位置，高亮显示相关要素
└── 期望: 清晰看到预警位置和周围环境

阶段4: 预警处理
├── 触发: 分析预警原因，决定处理方式
├── 操作: 调整航线、避让或确认忽略
└── 期望: 提供便捷的处理工具和操作入口

阶段5: 预警确认
├── 触发: 处理完成后确认预警
├── 操作: 点击确认按钮，记录处理结果
└── 期望: 预警状态更新，图标停止闪烁
```

### 12.2 航线规划流程

```
阶段1: 进入航线模式

├── 操作: 显示航线管理面板
└── 期望: 快速进入航线规划状态

├── 操作: 在海图上点击添加航点
└── 期望: 实时显示航线和航点信息

├── 触发: 拖拽航点 或 在属性面板修改参数
├── 操作: 调整航线形状和参数
└── 期望: 实时预览修改效果

阶段4: 检查航线
├── 触发: 点击"航线检查"按钮
├── 操作: 系统自动检测航线安全问题
└── 期望: 显示检查结果和改进建议

阶段5: 保存航线
├── 触发: 点击"保存"按钮
├── 操作: 保存航线到数据库
└── 期望: 航线保存成功，显示在航线列表中
```

#### 19.5.5 执行标准详细说明

##### 内存泄漏检测执行标准

**执行步骤**:
1. 启动应用，打开VisualVM，连接到应用进程
2. 执行典型操作流程：打开海图 → 缩放平移 → 切换面板 → 关闭面板 → 重复10次
3. 执行GC，记录堆内存使用量
4. 重复步骤2-3共5轮
5. 分析堆内存趋势

**工具使用**:
```bash
# 启动VisualVM
jvisualvm

# 或使用JProfiler
jprofiler.exe

# 命令行获取堆转储
jmap -dump:format=b,file=heap.hprof <pid>
```

**通过标准**:
| 指标 | 标准 | 判定方法 |
|------|------|----------|
| 堆内存趋势 | 稳定或下降 | 5轮操作后内存不持续增长 |
| 对象数量 | 稳定 | GC后对象数量恢复到基准值±10% |
| 监听器数量 | 不增长 | 每轮操作后监听器数量不增加 |

**不通过处理流程**:
1. 使用VisualVM分析堆转储，找出增长的对象类型
2. 检查对应代码中的监听器注册/移除逻辑
3. 修复后重新执行检测

##### 渲染性能测试执行标准

**执行步骤**:
1. 加载测试海图（包含10000+要素）
2. 启用帧率监控：`PerformanceMonitor.getInstance().enableFPSLogging()`
3. 执行缩放操作：放大→缩小→放大，持续30秒
4. 执行平移操作：上下左右平移，持续30秒
5. 记录平均帧率和最低帧率

**工具使用**:
```java
// 启用帧率监控
PerformanceMonitor monitor = PerformanceMonitor.getInstance();
monitor.enableFPSLogging(true);
monitor.setFPSLogInterval(1000); // 每秒记录一次

// 获取帧率数据
double avgFPS = monitor.getAverageFPS();
double minFPS = monitor.getMinFPS();
```

**通过标准**:
| 操作 | 平均帧率 | 最低帧率 | 判定方法 |
|------|----------|----------|----------|
| 缩放 | ≥30fps | ≥25fps | 30秒内平均值 |
| 平移 | ≥30fps | ≥25fps | 30秒内平均值 |
| 图层切换 | - | <100ms | 响应时间 |

### 22.2 技术实现可行性

### 22.3 关键技术点

| 技术点 | 方案 | 说明 |
|--------|------|------|
| 面板折叠动画 | Timeline + TranslateTransition | 平滑过渡效果 |
| 布局持久化 | Preferences API | 保存用户布局偏好 |
| 响应式布局 | ChangeListener + 绑定 | 窗口大小变化自适应 |
| 预警通知 | Notification + Animation | 非阻塞式提醒 |
| 面板拖拽 | SplitPane + ChangeListener | 左右面板宽度可拖拽调整 |

#### 22.3.0 设计决策说明

##### 1. 面板折叠动画方案选择

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **Timeline + TranslateTransition** | 平滑动画、可控性强、JavaFX原生支持 | 需要手动管理动画状态 | ✅ 采用 |
| CSS Transition | 简单易用 | JavaFX CSS动画支持有限 | ❌ 不采用 |
| 第三方动画库 | 功能丰富 | 增加依赖、体积增大 | ❌ 不采用 |

**风险与缓解**:
- 风险: 动画过程中用户快速操作可能导致状态不一致
- 缓解: 使用动画完成回调锁定用户操作，动画期间禁用相关控件

##### 2. 布局持久化方案选择

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **Java Preferences API** | 跨平台、无需额外依赖、系统级存储 | 存储格式受限 | ✅ 采用 |
| Properties文件 | 格式灵活、易于迁移 | 需要手动管理文件路径 | ❌ 不采用 |
| 数据库存储 | 支持复杂查询 | 增加系统复杂度 | ❌ 不采用 |

**风险与缓解**:
- 风险: 不同用户账户下布局偏好隔离
- 缓解: Preferences API自动按用户隔离，无需额外处理

##### 3. 响应式布局方案选择

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **ChangeListener + 属性绑定** | JavaFX原生、响应及时 | 需要手动管理监听器生命周期 | ✅ 采用 |
| 媒体查询(CSS) | 声明式、易于维护 | JavaFX CSS媒体查询支持有限 | ❌ 不采用 |
| 第三方响应式框架 | 功能完善 | 增加依赖 | ❌ 不采用 |

**风险与缓解**:
- 风险: 监听器未正确移除导致内存泄漏
- 缓解: 使用WeakChangeListener或确保在控件销毁时移除监听器

##### 4. 预警通知方案选择

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **Notification + Animation** | 非阻塞、可自定义样式 | 需要自行实现通知队列 | ✅ 采用 |
| Alert对话框 | 简单易用 | 阻塞用户操作、体验差 | ❌ 不采用 |
| 系统托盘通知 | 原生体验 | 跨平台兼容性问题 | ❌ 不采用 |

**风险与缓解**:
- 风险: 大量预警同时产生导致通知堆积
- 缓解: 实现通知队列，限制同时显示数量，旧通知自动消失

##### 5. 面板拖拽方案选择

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **SplitPane + ChangeListener** | JavaFX原生、自动处理分割线 | 自定义限制需要额外代码 | ✅ 采用 |
| 自定义拖拽实现 | 完全可控 | 实现复杂、容易出错 | ❌ 不采用 |
| DockFX框架 | 功能完善 | 项目维护不活跃、学习成本高 | ⚠️ 备选 |

**风险与缓解**:
- 风险: 用户拖拽到极端位置导致布局异常
- 缓解: 设置最小/最大宽度限制，超出范围自动调整

## 二十四、可行性评估

### 23.1 可行性结论

| 方面 | 评估 | 说明 |
|------|------|------|
| **技术可行性** | ✅ 高 | JavaFX组件完备，fxribbon已实现 |

### 23.2 最终推荐

采用方案B（右侧面板布局），理由如下：

1. **最适合海图显示系统**：海图显示区高度最大化，更符合海图浏览习惯
2. **预警系统友好**：右侧面板纵向展示预警信息，便于快速浏览和响应
3. **用户群体适配**：兼顾航海人员和专业管理员
4. **技术实现可行**：基于现有JavaFX技术栈，风险可控
5. **扩展性良好**：左右面板可独立扩展，适应未来功能需求
